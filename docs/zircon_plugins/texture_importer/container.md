# Texture Container Importer

The texture importer plugin owns lightweight header parsing for DDS, KTX1, KTX2, and ASTC container assets. It does not decompress those payloads. The importer emits `TextureAsset::new_container(...)` with the original bytes, a render-facing descriptor, mip count, array-layer count, and a normalized container format token such as `dds/dxgi-98`, `ktx/gl-internal-0x000083f1`, `ktx2/vk-37/supercompression-0`, or `astc/6x6x1`.

The production parser root lives in `container/mod.rs`; DDS-specific parsing lives in `container/dds.rs`, KTX entry wiring and shared KTX helpers live in `container/ktx/mod.rs`, KTX1 parsing lives in `container/ktx/ktx1.rs`, KTX2 parsing lives in `container/ktx/ktx2.rs`, KTX2 data-format-descriptor validation lives in `container/ktx/ktx2/dfd.rs`, ASTC parsing lives in `container/astc.rs`, and shared byte readers, parse diagnostics, layer-count checks, and container constants live in `container/support.rs`. Its regression fixtures and importer-facing tests live under the module-local `container/tests/` tree. `container/mod.rs` stays as the format dispatch and shared type boundary; `container/ktx/mod.rs` stays as the KTX-only entry/helper boundary; `common.rs` owns shared import fixtures and byte writers; and format-specific test files cover DDS, KTX1, KTX2, ASTC, descriptor settings, and cross-format diagnostics. DDS legacy header/pixel-format coverage is split into `container/tests/dds_header_pixel_format.rs`, KTX1 key/value metadata coverage is split into `container/tests/ktx1_metadata.rs`, and KTX2 key/value metadata plus metadata-range coverage is split into `container/tests/ktx2_metadata.rs`, so the format test files remain below the large-file threshold as more texture container boundaries are added. The shared test fixture module does not re-export runtime asset payload types; each format test imports `ImportedAsset`, `TexturePayload`, render dimension enums, and parser constants explicitly so dependency drift shows up in the owning test file instead of being hidden by broad module imports.

ASTC headers are validated before descriptor creation. The block dimensions and x/y/z extents must all be nonzero; malformed zero fields fail import with parse diagnostics instead of being silently normalized to one. Block dimensions must also match one of ASTC's supported 2D or 3D footprint tuples, such as `4x4x1`, `6x6x1`, `12x12x1`, `4x4x4`, or `6x6x6`; unsupported nonzero footprints fail before payload sizing. A 2D ASTC block footprint has `block_z == 1` and therefore requires the image depth field to stay `1`; a header that combines a 2D footprint with volume depth fails before payload sizing so it cannot produce a misleading D3 descriptor. After the header fields are read, the importer computes the declared compressed block payload as `ceil(width / block_x) * ceil(height / block_y) * ceil(depth / block_z) * 16` with checked arithmetic and requires those bytes to exist after the 16-byte ASTC header. If extreme 24-bit ASTC extents would make that block payload overflow the host range type, import fails with an ASTC payload-range diagnostic before any source length check or descriptor creation. A depth of one remains the valid 2D ASTC path, while depth greater than one or a 3D block depth marks the descriptor as `RenderImageDimension::D3`.

DDS DX10 cubemap arrays and KTX layer/face pairs multiply layer counts with overflow checks before metadata is emitted. DDS sources must contain the full 128-byte legacy header before any header bytes are read. After that length gate, DDS sources must begin with the literal `DDS ` magic before any DDS header fields are interpreted. DDS `DDS_HEADER.dwSize` must be 124 bytes before header fields are interpreted. DDS `dwFlags` must declare `DDSD_CAPS`, `DDSD_HEIGHT`, `DDSD_WIDTH`, and `DDSD_PIXELFORMAT`, matching the fields the importer reads for descriptor construction. `dwMipMapCount` only contributes to the emitted mip count when `DDSD_MIPMAPCOUNT` is set; without that flag the importer reports one mip level, and a declared zero mip count fails import. DDS containers recognized as block-compressed payloads must declare `DDSD_LINEARSIZE` and a nonzero `dwPitchOrLinearSize`; legacy uncompressed DDS containers without FourCC must declare `DDSD_PITCH` and a nonzero `dwPitchOrLinearSize`, so header flags describe the row or block-storage shape before payload bytes are preserved. Those two size flags are treated as mutually exclusive: compressed DDS payloads cannot also declare `DDSD_PITCH`, and uncompressed DDS payloads cannot also declare `DDSD_LINEARSIZE`, so one header cannot ask later upload code to interpret `dwPitchOrLinearSize` as both row pitch and total linear payload. A declared DDS mip count must also fit the parsed 2D extent's natural chain down to 1x1; over-large mip counts fail before payload sizing. DDS `dwCaps` must include `DDSCAPS_TEXTURE`; files missing the basic texture capability fail import before descriptor creation. DDS headers that declare multiple mip levels must also include both `DDSCAPS_MIPMAP` and `DDSCAPS_COMPLEX`, so a mip chain cannot be declared only through `dwMipMapCount` while the caps field describes a single-level texture. DDS cubemap headers must also include `DDSCAPS_COMPLEX` when cubemap face bits are present in `caps2` or when a DX10 header sets `DDS_RESOURCE_MISC_TEXTURECUBE`, keeping both cubemap declaration paths consistent before cube faces are expanded into array layers. A DX10 DDS header must not set both the legacy `DDSCAPS2_CUBEMAP` path and `DDS_RESOURCE_MISC_TEXTURECUBE`, because those paths give `arraySize` different meanings and would make the emitted array-layer count ambiguous. DDS `DDS_PIXELFORMAT.dwSize` must be 32 bytes before pixel format fields are interpreted, so malformed structures cannot shift the FourCC, bit-count, or channel-mask reads. DDS pixel format flags must agree with the FourCC field: nonzero FourCC bytes require `DDPF_FOURCC`, and `DDPF_FOURCC` requires a non-empty FourCC field. When the FourCC is `DX10`, the importer requires the full 148-byte DDS DX10 header before reading `dxgiFormat`, `resourceDimension`, `miscFlag`, `arraySize`, or `miscFlags2`. When no FourCC is present, the pixel format flags must declare exactly one primary uncompressed layout such as `DDPF_RGB`, `DDPF_LUMINANCE`, `DDPF_YUV`, `DDPF_ALPHA`, or `DDPF_BUMPDUDV`; `DDPF_ALPHAPIXELS` alone is treated as an auxiliary alpha-channel hint and is rejected without a layout flag, while multiple primary layout flags are rejected as ambiguous. For recognized block-compressed legacy DDS FourCCs and recognized block-compressed DDS DX10 `dxgiFormat` values, the importer walks every declared mip level, computes each level's block payload from `ceil(mip_width / 4) * ceil(mip_height / 4) * layer_count * bytes_per_block`, sums the mip chain, and requires those bytes after the 128-byte DDS header or 148-byte DDS DX10 header before emitting a container asset. That DX10 block table follows each BC family instead of treating numeric ranges as uniform: BC1 formats use 8-byte blocks, BC2/BC3 formats use 16-byte blocks, BC4 formats use 8-byte blocks, and BC5/BC6H/BC7 formats use 16-byte blocks. Typeless DXGI BC ids `70`, `73`, `76`, `79`, `82`, `94`, and `97` can still be preserved as container metadata when their payload length is structurally valid, but the runtime upload readiness layer must later reject them unless a concrete GPU format is chosen. DDS DX10 `dxgiFormat` must not be `DXGI_FORMAT_UNKNOWN`, because the importer cannot derive a stable texture format or upload contract from that value. DDS DX10 `arraySize` must be nonzero; a malformed zero value fails import instead of being normalized to one, so broken container headers do not produce misleading array metadata. Legacy DDS volume headers are also rejected before descriptor creation when `dwDepth` is nonzero or `DDSCAPS2_VOLUME` is set, because the current DDS path only has stable 2D, array, and cubemap descriptor contracts. DDS DX10 `resourceDimension` is currently accepted only for the texture-2D path that this importer can describe; unknown, zero, 1D, and 3D resource dimensions fail import until those descriptor paths are implemented. DDS cubemap face flags in `caps2` must not appear unless `DDSCAPS2_CUBEMAP` is also set, and DDS headers that set the legacy cubemap bit must include all six face bits; incomplete or face-only cubemap declarations fail import instead of being treated as ordinary 2D textures or complete six-face arrays. Nonzero DDS FourCC fields must be printable ASCII without embedded NUL bytes, so malformed tokens fail import before they can become lossy `dds/...` format strings. DX10 headers may also identify cubemaps through `miscFlag` `DDS_RESOURCE_MISC_TEXTURECUBE`; in that path, `arraySize` is interpreted as a cube count and the importer reports `arraySize * 6` array layers. Any other `miscFlag` bit is rejected because the importer has no descriptor contract for those options. DDS DX10 `miscFlags2` accepts only the documented alpha modes in the low three bits and requires all reserved upper bits to be zero. KTX face counts are constrained to ordinary textures or cubemaps: `1` is treated as one face, `6` is treated as a cubemap, and any other value fails import with a parse diagnostic. KTX1 and KTX2 declared mip counts must also fit the parsed width/height/depth natural mip chain down to 1x1x1; impossible KTX mip counts fail before metadata or level payload ranges are walked. Valid KTX1 and KTX2 3D containers keep native depth in `depth_or_array_layers` and force `array_layer_count` to one. KTX1/KTX2 headers that declare `depth > 0` must also declare a nonzero height, must not declare array layers, and must not declare cubemap face count `6`; malformed 3D extents, unsupported 3D array headers, and impossible 3D cubemap combinations fail import before descriptor creation. Cubemap KTX1/KTX2 headers must use nonzero-height square 2D faces, so face count `6` with height `0` or different width and height fails before descriptor creation.

DDS `DDSD_DEPTH` is treated as an unsupported volume declaration even when `dwDepth` is zero, so headers that advertise a depth field cannot slip through the current 2D/array/cubemap-only DDS path. Uncompressed DDS headers without FourCC must also provide a `dwRGBBitCount` in the 1..=32 range and at least one nonzero channel bit mask before the importer emits `dds/uncompressed`; a layout flag and pitch alone are not enough to describe the pixel storage. Channel masks must fit inside the declared `dwRGBBitCount` range, so a header cannot claim an 8-bit pixel while setting masks in higher bits. Channel masks must also be disjoint; the importer rejects headers where two channels claim the same pixel bit. When a header declares the `DDPF_ALPHA` layout flag or the auxiliary `DDPF_ALPHAPIXELS` channel flag, `dwABitMask` must be nonzero so alpha-bearing layouts identify where alpha bits live. When a header declares `DDPF_RGB`, at least one of `dwRBitMask`, `dwGBitMask`, or `dwBBitMask` must be nonzero so an RGB layout cannot be described by alpha bits alone. When a header declares `DDPF_LUMINANCE`, `dwRBitMask` must be nonzero because legacy DDS luminance layouts store their luminance channel in that mask slot. When a header declares `DDPF_YUV`, at least one of the first three channel masks must be nonzero so a YUV layout cannot be described by alpha bits alone. When a header declares `DDPF_BUMPDUDV`, at least one of the first three channel masks must be nonzero so a bump layout cannot be described by alpha bits alone.

KTX1 header scalar fields are validated before metadata and mip payload walking. `glTypeSize`, `glInternalFormat`, and `glBaseInternalFormat` must be nonzero, and `glTypeSize` must be one, two, or four bytes. `glType` and `glFormat` must also describe the same storage family: both zero for compressed data or both nonzero for uncompressed data. When both are zero, the source is the KTX1 compressed payload path and `glTypeSize` must be one. For uncompressed KTX1 headers, `glInternalFormat` must not equal `glFormat`, matching libktx's rejection of unsized internal-format headers. This keeps malformed KTX1 headers from producing a stable-looking `ktx/gl-internal-*` token when the source does not actually define a coherent OpenGL payload shape. KTX1 key/value metadata is not interpreted by the importer yet, but the declared byte range is still validated. The `bytesOfKeyValueData` field must be a multiple of four bytes and must fit inside the source bytes immediately after the 64-byte KTX1 header; unaligned or truncated files fail import before a descriptor or container payload is emitted. Inside that declared range, the importer walks each `keyAndValueByteSize` record, requires the size field, rejects zero-length records, requires the declared record payload to stay inside `bytesOfKeyValueData`, and advances over each record's four-byte padding. Each key/value record must also contain a non-empty NUL-terminated key whose key bytes are valid UTF-8 without a BOM. Multiple valid records are accepted by advancing over each padded record. Those record padding bytes must be zero, so malformed metadata cannot hide non-record bytes between valid records; malformed later records keep their record index in diagnostics. After that metadata range, every declared mip level must contain its `imageSize` field and any nonzero payload bytes. Intermediate levels require KTX1's four-byte mip padding range to exist and contain only zero bytes before the next level is read, so a header-only file, a malformed metadata record, a truncated first level, a nonzero inter-level padding byte, or a truncated later mip level cannot masquerade as a usable container payload.

KTX2 headers also keep basic scalar invariants explicit. `typeSize` must be nonzero before the importer reads extent, layer, face, and level-index fields, so malformed headers cannot proceed into descriptor creation with ambiguous element sizing.

KTX2 `supercompressionScheme` is validated before descriptor creation. The importer preserves official Khronos schemes `0` through `3` in the normalized format token, covering no supercompression, BasisLZ, Zstandard, and ZLIB. Reserved values and registered vendor/private schemes are rejected until Zircon has an explicit decoder or fallback contract for them, which keeps unknown bitstreams from entering the GPU-ready path as ordinary containers. BasisLZ KTX2 headers must declare `vkFormat` `0` because the concrete GPU format is selected by transcoding; a nonzero `vkFormat` with BasisLZ fails before level payloads or metadata are walked. When the scheme is `0`, `supercompressionGlobalData` must be empty; non-empty SGD bytes without an active supercompression scheme fail import so metadata from compressed layouts cannot be attached to an uncompressed container by accident. Zstandard and ZLIB KTX2 containers must also keep `supercompressionGlobalData` empty because those schemes do not define a global-data block; a non-empty SGD range with scheme `2` or `3` fails before metadata overlap checks. When the scheme is BasisLZ and any level payload is non-empty, `supercompressionGlobalData` must also be non-empty, so Basis universal data cannot enter the importer without its global endpoint metadata.

KTX2 metadata ranges are checked before the importer emits a texture asset. The importer currently keeps the original container bytes instead of decoding the data format descriptor, key/value data, or supercompression global data. The header-declared `dfdByteOffset + dfdByteLength`, `kvdByteOffset + kvdByteLength`, and `sgdByteOffset + sgdByteLength` ranges must fit inside the source file so truncated KTX2 metadata cannot masquerade as a valid container payload. DFD is mandatory for KTX2: `dfdByteOffset` and `dfdByteLength` must be nonzero, `dfdByteLength` must be at least 16 bytes, both fields must be 4-byte aligned, and the leading `dfdTotalSize` field must equal `dfdByteLength`. The DFD descriptor bytes after `dfdTotalSize` are walked as a descriptor-block chain. Each block must expose a zero vendor/type word, declare Khronos DFD version number `2` for the 1.3/1.4 block encoding, keep its transfer function field within the Khronos DFD enum range accepted by libktx, declare a block size of at least 24 bytes, fit inside the remaining DFD descriptor bytes, and remain aligned to the 16-byte sample-record stride after the 24-byte base block. The chain must consume the DFD range exactly, so malformed headers cannot leave trailing bytes or hide an oversized later block behind a valid first block. Empty KVD and SGD ranges must keep their offsets at zero, matching the KTX2 index convention that absent optional sections have no source location. Non-empty DFD, KVD, and SGD ranges must start after the header and level index table, and no two non-empty metadata ranges may overlap. Metadata ranges also cannot overlap any non-empty level payload range, so a header cannot alias descriptor bytes and texture image bytes to the same source region. Non-empty KVD must start at the next four-byte boundary after DFD. Non-empty SGD must start at the next eight-byte boundary after KVD when KVD exists, otherwise after DFD. Metadata ranges that point back into importer-owned structural bytes, alias another metadata or image block, place KVD away from the DFD-aligned location, or place SGD away from its aligned structure-following location are rejected with targeted parse diagnostics. When KVD is present, each key/value record must contain a four-byte `keyAndValueByteLength`, at least one key byte, a NUL key terminator, any value bytes, and zeroed four-byte padding that stays inside `kvdByteLength`; multiple valid records are accepted by advancing over each padded record. Keys must be valid UTF-8 without a BOM, and malformed later records keep their record index in diagnostics. The importer still does not interpret KVD values, but malformed record sizes, unterminated or empty keys, truncated payloads, and nonzero padding fail before descriptor creation.

Range validation is centralized in the parser helper path. KTX2 32-bit DFD/KVD fields are converted to the same 64-bit checked-add and overlap-guard path used by SGD ranges, and the level-index pass returns the occupied non-empty payload ranges for metadata alias checks. This keeps overflow handling, short-file diagnostics, metadata overlap rejection, payload overlap rejection, and structural-region rejection consistent across all KTX2 data blocks.

KTX2 level index entries are also checked for both byte ranges and uncompressed length invariants. After the level index table itself is proven present, each entry's `byteOffset + byteLength` payload range must fit inside the source bytes before descriptor creation. Non-empty level payloads must also start after the header and level index table, use an eight-byte-aligned `byteOffset`, and avoid every other non-empty level payload range, keeping importer diagnostics aligned with the runtime upload guard that rejects level payloads pointing back into metadata or into each other. Empty level payloads must keep both `byteOffset` and `uncompressedByteLength` at zero, so a level cannot point at a fake source location or claim decompressed bytes without any source payload. `uncompressedByteLength` must equal `byteLength` when `supercompressionScheme` is `0`, must be zero for BasisLZ, and must be nonzero for non-empty Zstandard or ZLIB payloads. For Zstandard and ZLIB, the declared uncompressed length must also be at least the compressed `byteLength`, so an inflate path never receives a level that claims to shrink during decompression. Any non-BasisLZ `uncompressedByteLength` must also be divisible by `faceCount * max(1, layerCount)` so later upload or inflate paths can split level payloads by image without guessing.
