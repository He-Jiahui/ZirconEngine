# UI Asset Documents And Editor Protocol

## Purpose

这份文档记录当前 shared UI 资产协议的三个硬结论：

- 生产序列化权威已经切到 tree-shaped `.ui.toml`
- formal loader 通过 production schema migrator 接受当前 tree、旧 schema tree、历史 flat node-table 和 legacy template fixture，并统一产出当前 tree authority
- editor/runtime 继续共用同一条 `UiAssetLoader -> UiDocumentCompiler -> UiTemplateSurfaceBuilder` 链路

本篇只描述“资产文档格式”和“editor/runtime 如何消费这份格式”。editor workbench owner 收口见 [`UI Asset Editor Host Session`](../editor-and-tooling/ui-asset-editor-host-session.md)。

## Serialized Authority Is Tree TOML

当前磁盘上的正式格式不再是：

- 顶层 `root = { node = ... }`
- 顶层 `nodes = { ... }` 作为长期权威注册表
- `components.*.root = "node_id"` 的根节点字符串引用

正式 authority 现在改成递归树：

- 顶层 `root` 直接内嵌根节点定义
- 每个节点显式带稳定 `node_id`
- 子节点通过 `children[].node` 递归内嵌
- 每条父子边的 `mount/slot` 元数据跟随 `children[]` 保存
- `components.*.root` 也直接内嵌子树，而不是再回指某个全局节点表 id

这意味着 `.ui.toml` 的 diff、迁移和 source roundtrip 已经围绕“真实树结构”展开，而不是继续把持久化权威建立在一张平面节点注册表上。

## Stable `node_id` Remains Mandatory

虽然磁盘格式改成了树，但 identity 没有改成“按位置寻址”。

当前仍然要求：

- 每个节点显式声明稳定 `node_id`
- source、hierarchy、preview、undo/replay 都以 `node_id` 作为跨表示层锚点
- 迁移器和 serializer 必须保住 `node_id`，而不是在输出时重新编号

这样 editor 侧的 selection、source cursor remap、document diff 和 preview projection 仍然能围绕稳定身份工作，而不会因为树形序列化就退化成位置敏感协议。

## Source Schema Version Policy

`asset.version` 现在被 `zircon_runtime::ui::template::asset::schema` 视为 source TOML schema version，而不是 compiled artifact 或 importer version。当前 source schema 版本固定为 `UI_ASSET_CURRENT_SOURCE_SCHEMA_VERSION = 3`，最低支持版本为 `1`。

版本策略是：

- `version == 3`：按当前 tree authority 解析和校验，migration report 记录 `CurrentTree` 与 `CurrentTreeValidated`。
- `version < 3`：迁到当前 source schema，写回 `asset.version = 3`，migration report 记录 `OlderTree` 和 `SourceVersionBumped`。
- 顶层 `[nodes.*]` flat node-table：归类为 `FlatNodeTable`，先 materialize 成递归 `root.children` / `components.*.root`，再写回当前 source schema。
- legacy `UiTemplateDocument` fixture：归类为 `LegacyTemplateFixture`，转换成 layout `UiAssetDocument`，只作为历史 fixture/importer 迁移路径存在。
- `version > 3`：返回 `UiAssetError::UnsupportedSchemaVersion { asset_id, version, current }`，不做静默降级；editor 后续可以基于这个结构化错误进入 read-only/save-as/downgrade 流程。

M7 只定义 source schema policy。M16 现在把 compiled artifact header、dependency manifest、runtime/editor stripping report 与 package validation 接到 compiler package validation surface；它仍然不把 `asset.version` 改成 binary artifact 或 importer version。

## Formal Loader Uses Schema Migrator

[`UiAssetLoader`](../../zircon_runtime/src/ui/template/asset/loader.rs) 不再直接 `toml::from_str::<UiAssetDocument>(...)` 后结束，而是统一走 [`UiAssetSchemaMigrator`](../../zircon_runtime/src/ui/template/asset/schema/migrator.rs)。

当前正式入口有两个层级：

- `UiAssetLoader::load_toml_str(...)` 返回当前 schema 的 `UiAssetDocument`，调用方无需知道 source 是否经历迁移。
- `UiAssetLoader::load_toml_str_with_migration_report(...)` 和 `UiAssetLoader::load_toml_file_with_migration_report(...)` 返回 `UiAssetMigrationOutcome { document, report }`，供 editor/importer 读取 source kind、版本 bump、flat materialization、legacy conversion 等结构化信息。

因此正式运行链路的内存 authority 仍然只有 tree，但 formal load path 可以接受受支持的旧 source 形态并在进入 compiler/surface 之前收敛成当前 tree authority。

## Flat And Legacy Migration Are Production-Owned

flat node-table 和 legacy template fixture 迁移已经从 `zircon_runtime/src/ui/tests/asset_migration_support.rs` 移到 production schema module：

- `flat_nodes.rs` 读取旧 `[nodes.*]` registry，按 root/component 引用 materialize 递归树，并检查 missing node / cycle 等源数据错误。
- `legacy_template.rs` 把旧 `UiTemplateDocument` fixture 转成 layout `UiAssetDocument`，保留 control id、bindings、attributes、slot mounts 和 component slot schema。
- `migrator.rs` 负责形态分类、future-version 拒绝、版本 bump、final `validate_tree_authority()` 和 report 组装。

这次收口后的真实边界是：

- 磁盘 canonical authority：tree TOML。
- 生产 loader：接受当前 tree、旧版本 tree、flat node-table、legacy template fixture，并统一输出当前 schema tree document。
- runtime test-only migration helper：已删除，不再作为 runtime 自己的迁移 owner。
- editor test support：仍可保留本地历史 fixture canonicalize，后续可按 editor active-session 边界再切到 production migrator。
- 生产内存访问：tree-native helper API。

## Editor Protocol Consequences

对 `zircon_editor::ui::asset_editor` 来说，这次 cutover 带来三条直接后果：

- canonical save 现在总是输出 tree TOML
- source roundtrip 仍然围绕稳定 `node_id`，不依赖旧平面节点表继续存在
- preview / undo / replay / style inspection / tree edit 已切到 `UiAssetDocument` 递归 helper，不再直接围绕 `document.nodes` 做权威更新
- editor production consumer 现在接收 loader 输出的当前 tree authority；旧 source 形态的迁移信息来自 `UiAssetMigrationReport`，而不是解析错误字符串

这也是为什么 editor 侧可以在不回滚 `UiAssetEditorSession` 既有 authoring 流程的前提下，完成 shared 资产真源切换，同时把 authoring 内部逻辑也收敛到 tree-shaped helper surface。

## Managed Asset Scope

这次被正式迁成 tree authority 的受管资产包括两类：

- editor/project 资产：`zircon_editor/assets/ui/**/*.ui.toml`
- runtime builtin fixture：`zircon_runtime/assets/ui/runtime/fixtures/*.ui.toml`

其中 runtime fixture 还额外完成了“加载入口不能放在 `src/` 下”的目录规范收口，详见 [`Runtime UI Graphics Integration`](../assets-and-rendering/runtime-ui-graphics-integration.md)。

## Shared Consumption Path

无论 editor 还是 runtime，现在都走同一条 shared 资产消费路径：

1. `UiAssetLoader` 读取 source TOML，并通过 `UiAssetSchemaMigrator` 收敛到当前 tree authority
2. `UiDocumentCompiler` 展开 import/component/slot/stylesheet
3. `UiTemplateSurfaceBuilder` 把编译结果落成 `UiSurface`

区别只在最后的宿主投影：

- editor 把 `UiSurface` 投影到 Rust-owned host contract
- runtime 把 `UiSurface.render_extract` 打包进 runtime frame / graphics pass

资产真源、编译器和 surface builder 没有 editor/runtime 分叉实现。

## Runtime Interface DTO Ownership Status

After the 2026-05-02 UI runtime-interface split, the canonical neutral DTO declarations for binding targets, binding expressions, binding reports, action-policy reports, localization reports, package validation reports, component contracts, resource references, schema reports, surface render records, event/control IDs, runtime binding values, component values, tree records, and layout geometry belong under `zircon_runtime_interface::ui`. Runtime modules below still own the concrete loader, schema migrator, compiler, validation, collection, cache, package, surface-builder, tree mutation/query behavior, and render-extraction behavior.

The M2 runtime hard-cutover now removes the runtime-local shadow modules for UI binding values, event control/reflection DTOs, component values, component descriptors, component data-source descriptors, template asset documents, template documents, resource/invalidation/action-policy/localization reports, package validation reports, schema migration reports, layout geometry/window DTOs, surface render DTOs, and UI tree/node DTOs. Runtime behavior imports neutral records from `zircon_runtime_interface::ui::*` and keeps behavior-only seams such as `UiAssetDocumentRuntimeExt`, `validate_component_descriptor(...)`, `extract_ui_render_tree(&UiTree)`, `virtual_window_for_scrollable_box(...)`, `UiRuntimeTreeAccessExt`, `UiRuntimeTreeRoutingExt`, `UiRuntimeTreeScrollExt`, `compiled_asset_header_from_cache_key(...)`, and `compiled_asset_dependency_manifest_from_imports(...)`. Editor/runtime projection code that needs template render commands calls `extract_ui_render_tree(&surface.tree)` instead of adding `UiRenderExtract::from_tree` behavior back onto the interface DTO. ## M15 Resource Reference Model

M15 当前 runtime/package 收口范围已经覆盖 typed model、resource dependency collector、compiled document dependency surface、M12 cache/invalidation input 和 M16 manifest consumption。`zircon_runtime::ui::template::asset::resource_ref` 仍是资源分类、fallback policy、collector 和 validation behavior owner；neutral resource DTO/report declarations 的 canonical owner 是 `zircon_runtime_interface::ui::template::asset`。editor 目前只消费这些 shared contract rows，不拥有资源分类、resolver policy 或 resource collection 规则。

显式资源依赖使用 typed table，而不是继续把所有资源藏在普通字符串里：

```toml
[[imports.resources]]
kind = "image"
uri = "res://textures/editor/folder.png"

[imports.resources.fallback]
mode = "placeholder"
uri = "res://textures/editor/missing-image.png"
```

`UiResourceKind` 当前固定为 `font`、`image`、`media`、`generic_asset`。`UiResourceFallbackPolicy` 当前固定为 `none`、`placeholder`、`optional`，其中 `none` 是默认 required-resource 语义，`placeholder` 必须提供 fallback URI，`optional` 表示资源缺失可接受。`UiResourceRef::validate(path)` 会用结构化 `UiResourceDiagnostic` 拒绝空 URI、不支持的 scheme、placeholder 缺少 URI、fallback 自引用和 placeholder fallback kind mismatch。M15 M1 接受的 URI scheme 是 `res://`、`asset://`、`project://`。

Legacy string inference 的基础规则放在 `UiResourceKind::infer_from_path_and_uri(path, uri)`：属性/source path 名称先于扩展名，`font/font_asset` 推断为 font，`image/icon/background_image` 推断为 image，`media/video/audio` 推断为 media，`asset/resource` 推断为 generic asset；扩展名再补充 font/image/media 分类，支持 `.font.toml`。没有更强分类但使用受支持 scheme 的资源 URI 默认归入 `GenericAsset`。

`UiResourceDependency` 和 `UiResourceDependencySource` 已作为 runtime-owned row type 被 `collect_document_resource_dependencies(...)` 使用。collector 会从 root document、当前 compiler 注册的 widget/style imports、显式 `imports.resources`、tokens、node props/params/layout、node style overrides、child slot metadata、stylesheet declaration blocks 和 component roots 中收集资源引用；字符串 URI 走 `UiResourceKind::infer_from_path_and_uri(...)`，typed TOML table 走严格 `UiResourceRef` validation。typed table classifier 只把含 supported resource `kind`、显式 `uri` 或 table-shaped `fallback` 的 TOML table 当作 resource table；因此 M14 localized text refs 的 string `fallback` 不会被误判为 resource fallback，layout/component metadata 里的 `kind = "VerticalBox"` 这类非资源表也会继续按普通 TOML table 递归扫描。这里的 import 范围刻意保持 package-input scope：M12 cache key 与 M16 widget/style dependency manifest 也以当前 registered import registry 为输入，而不是只记录本次展开实际触达的 import 子集。`UiCompiledDocument` 现在暴露 `resource_dependencies()` 和 `resource_diagnostics()`，使 package/editor/reporting consumer 不需要重新扫描 source。

M12 cache key 和 invalidation snapshot 现在包含 `resource_dependencies_revision`。`compile_with_cache(...)` 会先跑 document shape 与 component contract preconditions，再构建含资源 fingerprint 的 cache key，避免 invalid resource fingerprint error 掩盖更底层的 shape/contract 错误。资源依赖变化通过 `UiAssetChange::ResourceDependency` 映射到 `UiInvalidationStage::ResourceDependency`，并让 runtime render dirty 与 editor projection dirty，而不是触发无关的 document-shape 或 descriptor stage。M16 back-half 消费 compiled document 的完整 `resource_dependencies()` 写入 `UiCompiledAssetDependencyManifest.resource_dependencies`，所以 package manifest 没有第二套资源扫描规则。

剩余 productization 不在当前 M15/M16 收口内：resource diagnostics persistence beyond compile errors、resolver/file existence check、watcher/hot reload、editor read-only dependency view、resource browser、drag/drop picker、runtime loader backend 和 graphics/RHI consumption。

## M14 Localization Asset Metadata

M14 runtime foundation keeps localization collection and validation behavior in `zircon_runtime::ui::template::asset::localization`, while the neutral localized-text DTO/report declarations are canonical under `zircon_runtime_interface::ui::template::asset`. Authored values can use a TOML table with `text_key`, optional `table`, optional `fallback`, and optional `direction`; `UiLocalizedTextRef` validates the key, while `UiTextDirection` records `auto`, `left_to_right`, or `right_to_left` direction intent for runtime/editor consumers.

`collect_document_localization_report(...)` scans node props/layout/params and stylesheet declarations for localized tables, then records deterministic `UiLocalizationDependency` rows. Literal `text`, `label`, and `title` string values become `UiLocalizationTextCandidate` rows for extraction. `validate_document_localization(...)` runs during compiler preconditions, before component expansion, so invalid localization keys fail package validation without producing partial artifacts. Component prop validation treats a non-empty `{ text_key = ... }` table as a valid string-family prop while preserving the structured table in compiled attributes for package/report/runtime consumers. M14 does not resolve external locale tables, perform missing-key IO, or add editor locale preview UI in this slice.

## M21 Action Host Policy

M21 runtime foundation keeps action side-effect classification and host-policy validation behavior in `zircon_runtime::ui::template::asset::action_policy`, while the neutral policy/report DTO declarations are canonical under `zircon_runtime_interface::ui::template::asset`. `UiActionSideEffectClass` classifies existing `UiBindingRef` route/action strings into local UI, editor mutation, asset IO, scene mutation, external process, or network classes, so this does not create a second action path next to the existing binding model.

Package validation now emits `UiActionPolicyReport`. Runtime profile uses `UiActionHostPolicy::runtime_default()` and only allows local UI actions; editor profile uses `UiActionHostPolicy::editor_authoring()` and also allows editor mutation and asset IO. Unsafe side effects remain diagnostics in the package report for host enforcement/inspection; editor policy-inspector UX remains future work, while M18 binding-expression semantics now lives in the runtime-owned binding module described below.

## M18 Binding Expression Semantics

M18 keeps `zircon_runtime::ui::template::asset::binding` as the runtime-owned semantic validation layer for authored binding targets and restricted expressions, while neutral binding target/expression/report DTO declarations are canonical under `zircon_runtime_interface::ui::template::asset`. The runtime binding root now imports those interface DTOs directly and retains only validation behavior plus structural exports. Existing `UiBindingRef` keeps event, route, and action payload authority, and now carries optional `targets: Vec<UiBindingTargetAssignment>` so a binding can describe prop, class, visibility, enabled, or action-payload assignments without creating an editor-only schema fork.

Serialized targets use compact TOML tables. The target `kind` values are `prop`, `class`, `visibility`, `enabled`, and `action_payload`; `prop`, `class`, and `action_payload` require `name`, while `visibility` and `enabled` are node-level boolean targets. Expressions are authored as strings and parsed by runtime into `UiBindingExpression` for static validation. The supported language is deliberately small: bool/int/float/string/null literals, `param.<name>`, `prop.<name>`, `==`, `!=`, `&&`, `||`, `!`, and parentheses. Runtime does not execute host actions, call arbitrary functions, mutate state, or evaluate general scripts in M18.

`collect_asset_binding_report(document, registry)` walks root and component node trees and emits structured `UiBindingDiagnostic` rows with codes `InvalidTarget`, `InvalidValueKind`, `UnresolvedRef`, and `UnsupportedOperator`. `validate_asset_bindings(...)` converts the first error into `UiAssetError::InvalidDocument` for existing compiler callers. `UiDocumentCompiler::validate_compiler_preconditions(...)` now runs binding validation after shape/localization/component-contract validation and before expansion/cache-key construction, so malformed binding semantics cannot be masked by later package, resource, or style behavior.

When a runtime descriptor exists for the node type, descriptor props are authoritative for both prop targets and `prop.<name>` references; authored node props are only a fallback for descriptor-less nodes. Action-payload targets must name an existing payload field on the binding action instead of silently accepting unknown keys as `Any`. The parser classifies unsupported or malformed operators such as single `=`, `&`, and `|` as `UnsupportedOperator` diagnostics.

Editor preview helper functions such as `=concat(...)`, `=get(...)`, and node-path mock expressions remain editor-preview behavior. Runtime validates M18-owned expressions in target assignments, and it validates action payload strings that begin with `=` only when they use the M18 runtime expression prefix forms such as `prop.` or `param.`. This preserves current editor preview fixtures while moving the accepted runtime binding target schema into `zircon_runtime`.

Editor projection now consumes the same runtime binding report when preview compilation fails. `diagnostics/binding.rs` maps the interface-owned `UiBindingDiagnostic` into `UiAssetEditorDiagnostic`, preserving code, severity, source path, target node, and `target_binding_id`. `binding/schema_projection.rs` renders authored target assignment rows and appends matching target diagnostic rows by target index; it does not infer value kinds independently or create a parallel editor binding semantic model. ## M13 Descriptor-Driven Palette

M13 把现有 Runtime UI component catalog 收口成 `zircon_runtime::ui::component` 下的 descriptor registry authority。descriptor 现在声明 host capability、render capability、fallback policy、palette metadata 和 default node template，并在 registry registration 时执行结构化 validation。registry revision 只在已注册 descriptor set 发生实际变化时递增，供后续 compiler cache/invalidation 使用。

Editor asset palette 不再维护硬编码 native widget 清单。`zircon_editor::ui::asset_editor::palette` 从 `UiComponentDescriptorRegistry::editor_showcase().palette_entries_for_host(&UiHostCapabilitySet::editor_authoring())` 生成 native entries，然后继续追加当前 document 的 `components.keys()` 和已导入 `widget_imports.keys()`，这两类仍然是非 descriptor palette source。

Default native insertion 使用 runtime descriptor 提供的 `UiDefaultNodeTemplate`，因此 Button/Label/TextField 这类默认 props、layout container 这类默认 layout、以及未来 widget capability metadata 都从 shared runtime authority 进入 editor authoring flow。

## M10 Component Public Contracts

M10 在 shared asset document 协议里补上 component public contract，而不是让 editor 或 compiler 继续靠约定访问内部节点。`UiComponentDefinition.contract` 描述 component 的 exported API version、public parts、root class policy、focus contract 和 binding route contract；`UiNodeDefinition.component_api_version` 描述引用方需要的组件 API 版本。

默认 contract 是闭合的：外部文档可以引用 component root、传参、填 slot，并通过显式导出的 `:part(...)` 选择器访问 public part；不能直接用内部 `node_id` 或 `control_id` 作为 stylesheet selector 触达 imported widget internals。编译器在 shape validation 后、component expansion 前执行 contract validation，这样 private-boundary 或 API 版本问题不会等到展开后泄露内部结构。

M10 foundation gate 后续的第一块 productization 已经落地：`UiComponentContractDiagnostic` 和 `UiComponentContractDiagnosticCode` 由 runtime component-contract validation 产生，覆盖 invalid public part、private selector、API mismatch、closed root class、private binding target 和 private focus target。compiler 主路径仍然把首个 contract diagnostic 转成原有 `UiAssetError::InvalidDocument`，所以既有 compiler 调用方不需要解析新 DTO；需要作者态定位的 editor 可以直接调用 `component_contract_diagnostic(...)` 消费 runtime-owned code/path/target metadata。

Editor 侧只做投影：`UiAssetEditorDiagnostic` 保存 code、severity、message、source path、target node/control/binding id；`UiAssetEditorSession` 继续保留原有 `diagnostics: Vec<String>` 给旧 presentation/status 行，同时维护 `structured_diagnostics`。当 runtime diagnostic 提供 `target_node_id` 时，pane presentation 会优先把 source outline selection 指向对应节点；diagnostic panel 行则使用稳定格式 `severity [code] source_path: message`。

M10 product-facing follow-up also now covers root-class authoring UX: the UI Asset Editor projects `UiRootClassPolicy` for the selected local component, exposes `append_only` / `closed`, and commits edits through the Runtime UI `component.root_class_policy` adapter path so runtime component-contract validation remains the only semantic authority. M18 runtime binding-expression semantics is now accepted separately under `zircon_runtime::ui::template::asset::binding`; editor diagnostic projection now consumes that runtime report for structured binding diagnostics and target-schema inspector rows.

## M12 Invalidation And Compile Cache

M12 在 shared asset/template 层新增 `UiInvalidationGraph` 和 `UiAssetCompileCache`，用来描述“为什么需要重新编译”而不是只靠 editor 局部 dirty flags 猜测。cache key 包含 root document fingerprint、widget/style import fingerprints、descriptor registry revision 和 component contract revision；这些输入覆盖 M7 schema authority、M13 descriptor authority 与 M10 contract authority。

`UiDocumentCompiler::compile_with_cache(...)` 是现阶段的正式入口：key 完全一致时返回 cache hit；key miss 时保留现有 full compile 行为，同时返回 `UiInvalidationReport`，说明 document/import/descriptor/contract 哪些输入变化触发了 SourceParse、DocumentShape、ImportGraph、DescriptorRegistry、ComponentContract、SelectorMatch、StyleValue、Layout、Render、Interaction、Projection 等阶段。cache 的 last-snapshot 记录按 asset kind/id 分桶，而不是使用一个全局 last snapshot；这样交错编译多个 document 时，miss report 只会和同一个 asset 的前一版比较。

M12 还把 invalidation stage 映射回 runtime `UiDirtyFlags`，并用 named thresholds 生成 broad selector 与非虚拟化 `ScrollableBox` 大子节点诊断。持久化缓存 artifact 与 editor 侧性能可视化仍是后续产品化工作；package/build 报告入口已由 M16 接入。

## M16 Package Validation Report

M16 在 `zircon_runtime::ui::template::asset::compiler::package` 下新增 package validation owner。`UiDocumentCompiler::validate_package(document, profile)` 先执行和 full compile 一致的 document shape 与 component contract preconditions，再通过 runtime `compile_cache_key_from_compiler(...)` 构建 interface-owned `UiCompileCacheKey`，随后走 full `compile(...)` 路径触发 descriptor/default prop、stylesheet 与 resource dependency validation；因此 M16 不绕开 M10/M12/M13 的规则，也不会新增另一套 import 或 contract 判断。

`UiCompiledAssetHeader` 记录 source schema version、compiler/package schema version、descriptor registry revision、component contract revision、root document fingerprint 与完整 compile cache key。`UiCompiledAssetDependencyManifest` 用 deterministic order 输出当前 compiler 注册的 widget/style imports、asset id、asset kind、source schema version 与 fingerprint，使 package diagnostics 能看见 M12 cache key 的真实输入；M16 back-half 还把 compiled document 的完整 M15 package-input resource dependency collector output 作为 `resource_dependencies` 写入同一个 manifest surface。M14 继续在同一个 manifest surface 写入 `localization_dependencies`，避免 package manifest 另起一套 localization scan。

`UiCompiledAssetPackageValidationReport` 记录 `UiCompiledAssetPackageProfile::{Runtime, Editor}`，并用 retained/stripped sections 表达 profile 差异。Runtime profile 保留 runtime template tree/style values/bindings，剥离 source document、authoring diagnostics 与 migration report；Editor profile 保留两侧信息。Package report now also carries M21 `action_policy_report` and M14 `localization_report`, so host policy diagnostics, locale dependencies, literal extraction candidates, and localization diagnostics remain runtime-owned metadata.

M16 back-half 在同一 package owner 下新增 binary artifact 与 package manifest surface。`UiDocumentCompiler::compile_package_artifact(...)` 返回 runtime-owned `UiRuntimeCompiledAssetArtifact { report, compiled }`，`UiRuntimeCompiledAssetArtifact::to_bytes()` 写入稳定 magic `ZRUIA016`、little-endian `u32` binary artifact schema version 和 little-endian `u64` payload length，再保存 UTF-8 TOML payload。payload 仍是 typed `UiRuntimeCompiledAssetArtifact { report, compiled: UiTemplateInstance }`，但避免对 `toml::Value` compiled payload 使用 `bincode`。`from_bytes(...)` 校验 envelope 后恢复 runtime artifact。Runtime-interface cutover 后，`UiCompiledAssetArtifact` 这个名称只表示 interface-owned neutral DTO `{ report, bytes }`，而 runtime 的 compiled template-instance payload 使用 `UiRuntimeCompiledAssetArtifact`，避免同名不同语义。`UiCompileCacheKey`、`UiCompiledAssetCacheRecord`、`UiCompiledAssetPackageManifest` 与 manifest entry/header/dependency DTOs 由 `zircon_runtime_interface::ui::template` canonical owner 提供；runtime 只保留 `compile_cache_key_from_compiler(...)` 和 `compiled_asset_package_manifest_from_artifact_bytes(...)` 这类需要 compiler state 或 runtime artifact payload 的行为 helpers。`UiCompiledAssetCacheRecord` 继续记录 header、完整 M12 cache key、由 key 生成的 `UiInvalidationSnapshot`、artifact fingerprint 和 byte length，用作 future persistent cache index。`UiCompiledAssetPackageManifest` 用 TOML writer/importer 保存 header、dependency manifest、cache record 和 artifact entry；它消费同一个 `resource_dependencies` / `localization_dependencies` manifest surface，不创建第二套 resource 或 localization list。
