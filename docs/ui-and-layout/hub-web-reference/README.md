# Zircon Hub Web Reference

This folder contains the HTML/CSS-finalized Hub visual reference pages.
`docs/ui-and-layout/hub.png` remains the fixed Projects Dashboard source
reference. The other Hub design PNGs are exported from this static page set.
The old `tools/generate-hub-design-assets.py` generator is retired; final Hub
design PNGs are owned by this AI-directed, HTML/CSS-finalized workflow.

## Files

- `index.html` is the browser-openable multi-page reference shell.
- `app.js` owns the page data, route selection, overlays, state rendering, and
  primary `data-route` routing for the main reference flows.
- `cover-rendering.js` maps project ids to local reference cover images consumed
  by `app.js`.
- `styles.css` owns the Hub visual tokens, layout, and component density.
- `covers.css` owns project cover image treatment. Project cards, project table
  thumbnails, browser rows, side previews, and detail hero previews use the same
  local concept-art cover family as the fixed `hub.png` reference.
- `responsive.css` owns live-window responsive overrides. It preserves the
  `1568x1003` export baseline while making browser preview layouts fit smaller
  desktop windows without document-level overflow.
- `fullscreen-preview.css` makes normal browser preview fill the viewport at
  every checked size; the export still captures the same `1568x1003` canvas.
- `interaction-states.css` owns opt-in visual states for local shell controls,
  search filtering, and button-state demos without changing the default export.
- `interaction-enhancements.js` owns local UI feedback, fallback action routing,
  and live filtering for controls that are not direct `data-route` links.
- `LAYOUT_MODEL.md` records the CSS region model, breakpoint model, runtime
  asset policy, and Rust-window migration notes.
- `page-registry.mjs` is the shared source of truth for the canvas size,
  dashboard capture, and 19 exported page ids.
- `export-pages.mjs` exports the final `1568x1003` PNG set with Playwright.
- `validate-visuals.mjs` checks exported PNG dimensions/content and compares
  the dashboard capture against `docs/ui-and-layout/hub.png`, while also
  checking `EXPORTS.md` and the AI reference manifest against the shared page
  registry.
- `validate-interactions.mjs` checks browser-openable click routes with
  Microsoft Edge's debugging protocol, fails if any authored `data-route`
  target or `href="?page=..."` link target is not a known page id from
  `page-registry.mjs`, and rejects actionable buttons without a route or local
  UI handler.
- `validate-responsive.mjs` checks all browser-openable pages across seven
  viewport sizes using DOM geometry instead of screenshots, then resizes
  representative pages in-place through five live resize steps. It also
  validates local-only runtime dependencies and rejects UI reuse of AI draft
  PNGs, exported Hub PNGs, or raster project cover thumbnails.
- `hub-web-reference-1568x1003.png` is the captured dashboard reference page.
- `EXPORTS.md` is regenerated by the export script and lists each exported
  PNG, its page id, and the matching `index.html?page=<page_id>` replay path.
- `SPOT_CHECKS.md` records the representative manual visual inspection set for
  dashboard, Editor, Assets, Project Detail delete confirmation, Source Engine
  popup, Empty, and Error review.
- `ACCEPTANCE_EVIDENCE.md` records the export command, generated final PNG
  inventory, validation commands, focused Cargo contract result, and known
  acceptance limits.
- `../hub-ai-drafts/*.png` contains structure-layout direction drafts for
  reviewing overall page structure before the HTML/CSS reference is finalized.
  The directory must contain exactly the 19 manifest-listed PNGs, with no orphaned AI draft PNGs or stray draft files.
- `../hub-ai-reference-manifest.schema.json` defines the required manifest
  fields, 19 reference rows, 3 supplemental design artifacts, and acceptance
  evidence links.

## Workflow

1. Review or adjust the AI prompt manifest at
   `docs/ui-and-layout/hub-ai-reference-manifest.json`.
2. Open `index.html?page=hub-editor` or any manifest `page_id` in a browser.
   The shell also accepts exported PNG filenames such as
   `index.html?page=hub-editor.png` and normalizes them to the matching page id.
3. Run `node docs/ui-and-layout/hub-web-reference/export-pages.mjs`.
4. Run `node docs/ui-and-layout/hub-web-reference/validate-visuals.mjs`.
5. Run `node docs/ui-and-layout/hub-web-reference/validate-responsive.mjs`.
6. Inspect the resulting `docs/ui-and-layout/hub-*.png` files.
7. Run `node docs/ui-and-layout/hub-web-reference/validate-interactions.mjs`
   after changing click routing.

Use `node docs/ui-and-layout/hub-web-reference/export-pages.mjs --only=<page_id>`
to refresh selected design PNGs while still updating the dashboard web capture.
The generated `EXPORTS.md` rows can then be opened directly from the replay
path in parentheses. Unknown `--only` page ids fail before capture starts, and
the error lists the available page ids from `page-registry.mjs`.
The exporter starts on port `5198` by default and automatically falls back to a
free local port when that default is already occupied; an explicit
`ZIRCON_HUB_WEB_REFERENCE_PORT` remains strict.

The static shell is also clickable: the source-engine selector, user menu,
sidebar navigation, window/collapse controls, `New Project`, project cards,
filter/sort controls, list view, Quick Actions, page action rows, confirmation
buttons, pagination, button-state demos, and search inputs either route to their
matching reference pages or provide local UI feedback.
The shell also records the normalized `data-page` on `.hub-shell`, so validation
can verify output-filename replay paths without relying on sidebar active state.

For design-review mode, use `docs/ui-and-layout/hub-design-board/`. Those
boards are separate annotated structure/function diagrams and are not part of
the 19 final web-reference PNG exports. The structure board and structure
supplement are the primary artifacts for overall layout review; the functional
details board is secondary.

The AI direction prompts are tracked as design intent. Their review target is
overall interaction structure: navigation, page regions, toolbar placement,
card/table grids, side panels, dropdowns, modals, and state overlays. Local
functional-content callouts are secondary. Final text, icons, and component
geometry are controlled by HTML/CSS so the reference images remain usable as
precise UI alignment targets.
Runtime preview does not depend on third-party UI libraries or screenshot
composition. The shell, controls, panels, state surfaces, menus, and overlays
are authored HTML/CSS; raster images are limited to brand, icon, and local
project cover assets.
