# Hub actionable pages

This document owns the page action slice that replaced the broad `hub-pages` milestone. The parent milestone remains intentionally decomposed in workflow planning, while this page records the shared behavior now expected from Builds, Editor, Assets, Plugins, Learn, Team, Cloud, and Settings before the later integrated-journeys milestone connects complete end-to-end flows.

## Page scope rules

All workspace pages derive project context from the canonical Hub snapshot rather than duplicating page-local lookup rules. Selected-project actions require the selected path to still resolve through the recent-project registry. If the selected path is stale, actions report a targeted recovery message instead of silently falling back to the latest recent project. Latest-recent fallback is reserved for dashboard-style quick actions when no explicit selected project exists.

Source Engine context follows the same rule. A selected project with a bound engine uses that engine; an unbound or unavailable selected-project engine remains visible as an unready state and must not borrow the global active-engine fallback. Active or first-engine fallback only applies to source-level views that have no selected-project context.

## Builds and Editor pages

The Builds and Editor pages surface selected-project title, path, readiness, Source Engine context, recent source-build history, and operation timeline rows. `BuildsPage` forwards build, launch, package, and install callbacks through selected-project-only actions declared on `HubWindow`, so page actions cannot accidentally reuse Dashboard quick-action fallback. The component modules keep action labels and disabled details visible even when the action cannot run.

`view_model/workspace_actions.rs` is the shared lower projection for those page-specific children. It emits `WorkspaceActionReadinessData` through `HubWindow.workspace-action-readiness`, keeping one Rust model for selected-project title/path/status, project-bound versus active Source Engine state, build/open-output/open-editor/package/install booleans, disabled reasons, the current source-build summary, and the latest operation summary. The model treats `ProjectScope::LatestRecent` as unavailable for page actions even though Quick Actions may use it, reports stale selected projects with action-specific text, blocks build on unbound or unavailable project engines, blocks editor launch when the staged editor executable is missing, and reports output/device path gaps separately from project or engine readiness.

The Builds page now consumes that readiness model directly instead of deriving button state from `ProjectDetailData.can_*` fallback copy. `HubWindow` passes `workspace-action-readiness` into `BuildsPage`, `BuildControlsPanel` uses the readiness booleans and disabled-reason strings for Build Editor, Open Output, Open Editor, Package Project, and Install To Device rows, and the source summary/current task areas show the model-provided source-build and operation summaries. The build pipeline rows also use the same source/project readiness for their status badges and disabled details, keeping the Builds surface selected-project-only while still forwarding callbacks through `build-selected-project-engine`, `launch-selected-project`, `package-selected-project`, and `install-selected-project`.

The Editor page now consumes the same readiness model for its source summary and action panel. `HubWindow` passes `workspace-action-readiness` into `EditorPage`; `EditorSourceSummaryPanel` titles the page around the selected project, Source Engine title/status, selected-project status, and source-build summary instead of showing only the active engine. `EditorActionsPanel` keeps Save Source available for Source Engine settings, but Build, Open Output, and Open Editor rows now use the readiness booleans and disabled-reason strings. Editor-page Build and Open Editor callbacks route through `build-selected-project-engine` and `launch-selected-project`, so the Editor page cannot use the Dashboard quick-action latest-recent fallback when no selected project exists or when the selection is stale. The registered Source Engine list and source settings fields remain source-management controls and keep their existing select, rename, browse, remove, and save callbacks.

Runtime selected-project helpers in `project_workspace.rs` provide the action target and bound Source Engine gates. Early failures such as no selected project, stale selection, missing project root, or missing bound Source Engine are recorded through the same action-history model as process-spawn failures, preserving recovery text and status-bar scope.

The Builds/Editor contract gate locks both static page wiring and runtime summary projection. `ui_selected_project_runtime_contract.rs` now requires the Builds and Editor Slint snippets to keep `WorkspaceActionReadinessData`, source-build summaries, operation summaries, disabled-reason strings, and selected-project-only callbacks in place. `workspace_action_readiness` unit coverage also verifies that empty source-build details fall back to the build status, empty action-history details fall back to the action label, missing output roots disable Open Output and Package with `Configure a build output directory`, and missing device-install roots disable Install with `Configure a device install directory`. This keeps the UI contract tied to executable Rust projection behavior instead of only checking string presence in Slint files.

## Catalog pages

Assets, Plugins, and Learn use source-scoped catalog refresh paths but project selected-project entries first. Runtime catalog modules discover selected-project assets, plugin manifests, and local learning docs before merging Source Engine/global entries. View-model projection labels each row's scope and exposes empty-state copy that distinguishes selected-project catalogs from broader Source Engine catalogs.

`catalog_page_components.slint` owns the shared catalog row wrappers so `assets.slint`, `plugins.slint`, and `learn.slint` remain focused on page titles, row actions, selected/global copy, row counts, and Learn open-resource routing. Static contracts keep those page roots selected-project aware without letting each page recreate layout behavior.

The Learn open-resource action is intentionally narrower than generic file opening. `LearnRow` can request an open action only with its projected catalog path; `HubRuntime::open_learn_resource` re-resolves that path against the current `learn_catalog` using the shared normalized filesystem key, rejects empty paths, rejects paths that are not present in the refreshed catalog, and rejects catalog entries whose files disappeared before launch. This keeps the only row-level catalog action tied to concrete selected-project or Source Engine documentation discovered by the scanner, while Assets and Plugins remain read-only until real asset/plugin actions exist.

## Team and Cloud pages

The Team page projects local repository and identity state for the selected project when available, then falls back to Source Engine repository context only when no selected-project repository is available. Missing Git identity, no repository, and source fallback states stay explicit so the page remains useful in offline/local-only mode.

The Cloud page is intentionally local package/install orchestration rather than a remote account integration. It reports package output and local device-install status for the selected project, counts only selected-project package and install records when a project is selected, and disables package/install actions with specific no-selection, stale-selection, or unready reasons. Package readiness requires both a selected project that still resolves through `HubScope` and a configured package output root; install readiness uses the same selected-project gate plus a configured device-install directory. The disabled detail names the missing local root instead of exposing the later runtime failure text, and stale selections report that the selected path no longer exists in the recent-project registry. Its action panel routes only through selected-project callbacks; unlike Dashboard quick actions it must not promote the latest recent project when no selected project is present.

Cloud also embeds the persisted operation timeline for package and install outcomes. Successful package/install records and recoverable failures are projected from `HubActionRecord` into the Cloud page so users can see the last local package path, device install path, recovery hint, or stale-selection failure without switching to Builds or Settings.

## Settings page

Settings projects configuration as status rows instead of static descriptive copy. The page summarizes toolchain paths, default project root, Source Engine checkout and output directories, build profile/jobs, device install root, selected-project context, and save/register results. Invalid paths and missing Source Engine state are warning or disabled states, not hidden configuration details.

`SettingStatusData` rows now include scope, action id, action label, disabled reason, and actionable state. `view_model::settings_statuses` derives those rows from the full `HubSnapshot`, so Settings can explain stale selections, latest-recent fallback, project-bound Source Engines, active Source Engines, unbound projects, unavailable engine ids, and missing Source checkout paths. Row actions route through the existing Settings callbacks: save/register stays on `save-settings`, while path repair affordances reuse the same folder-picker targets as the editable path fields.

Settings also shares operation timeline projection with Builds. This keeps persisted build, package, install, editor-launch, and open-output failures visible from the configuration surface where users are most likely to repair paths.

The parent `hub-pages` plan was too broad for one shallow executor slice, so the workflow split it into four implementation children plus a contracts/docs gate. The current contract surface verifies the resulting page behavior across selected-project runtime actions, selected-project catalog copy, page surface coverage, quick-action fallback, and Hub package tests.

`hub-docs-contract-refresh` keeps the actionable page rules aligned with the current component and runtime contracts. Builds and Editor consume `WorkspaceActionReadinessData` for selected-project-only actions, disabled reasons, source-build summaries, and operation summaries. Builds, Settings, Cloud, and related panels share `OperationTimelinePanel` for persisted action history, while Dashboard Quick Actions remain the only latest-recent fallback surface.

The handoff is intentionally documentation-only: it does not add a new asset/plugin action model, remote Cloud account lifecycle, or new build/install process path. `hub_docs_contract.rs` scans this document so the next `hub-acceptance-validation` milestone can use the recorded selected-project-only page behavior and timeline ownership instead of re-auditing every page from scratch.
