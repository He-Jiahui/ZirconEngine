# Runtime Sound Extension

## Purpose

`sound` is now documented as the first core slice of a plugin-owned audio engine, not only as the earlier WAV clip playback loop. The stable ownership remains the same:

- `zircon_runtime::core::framework::sound` owns neutral DTOs, IDs, errors, graph descriptions, component descriptors, automation contracts, plugin options, and the `SoundManager` trait.
- `zircon_plugins/sound/runtime` owns concrete state, clip/source playback, mixer graph validation, CPU DSP, meters, static impulse responses, deterministic software output, and the feature-gated CPAL output adapter.
- `zircon_plugins/sound/editor` owns editor contribution descriptors for mixer authoring, acoustic debug, sound component drawers, and the editor-facing live output snapshot/action model over the neutral `SoundManager`.

The design follows the repository boundary rule: heavy audio behavior stays in the independent sound plugin, while `zircon_runtime` exposes only neutral contracts and host-facing registry surfaces.

## Related Files

The sound contract is split by responsibility:

- `ids.rs` defines stable IDs for clips, playbacks, tracks, nodes, effects, sources, listeners, volumes, parameters, automation bindings, impulse responses, and external source handles.
- `graph.rs` defines `SoundMixerGraph`, tracks, sends, controls, meters, and snapshots.
- `acoustics.rs` defines ray-tracing convolution status, neutral HRTF profile descriptors, and the neutral ray-traced impulse-response submission descriptor used by optional geometry/ray-query providers.
- `effects.rs` defines the first deterministic DSP descriptor set: gain, filter, reverb, convolution reverb, compressor with sidechain, wave shaper, flanger, phaser, chorus, delay, pan/stereo, and limiter.
- `components.rs` defines neutral `AudioSource`, `AudioListener`, `AudioVolume`, and external source block contracts plus plugin dynamic component type IDs. Source/listener descriptors now carry position, forward vector, velocity, gain, speed, playing, loop, mute, start/duration, and spatial-scale data so the plugin runtime can spatialize and apply Bevy-style source playback controls without coupling directly to a concrete scene transform component.
- Framework `automation.rs` defines Timeline-style sound automation bindings, serializable scalar automation curves/keyframes/interpolation modes, and neutral `SoundTimelineSequence` scheduling DTOs for binding curves to sound automation targets over time.
- Framework `events.rs` defines the versioned dynamic event catalog, descriptors, handler descriptors, opaque runtime invocation payloads, deterministic delivery DTOs, and neutral execution reports. Sound owns handler registration and queue fan-out mechanics while event meaning and side effects remain in the consuming plugins.
- `SoundManager::global_volume_gain` and `SoundManager::set_global_volume_gain` expose the plugin's master gain as a neutral Bevy `GlobalVolume`-style runtime control. The concrete runtime stores this on `SoundConfig::master_gain` and applies it after mixer graph rendering, so clip playbacks, sources, sends, and effects share the same final output scalar. The package manifest also exposes `sound.global_volume_gain` so profile/export/editor surfaces can show the default value instead of hiding it inside runtime config.
- `SoundManager::default_spatial_scale` and `SoundManager::set_default_spatial_scale` expose the Bevy `DefaultSpatialScale` concept from `dev/bevy/crates/bevy_audio/src/audio.rs` and `audio_output.rs` as a neutral scalar. `SoundSpatialSourceSettings::spatial_scale` mirrors Bevy `PlaybackSettings::spatial_scale` as a per-source override. The concrete runtime stores the default on `SoundConfig::default_spatial_scale`, exposes `sound.default_spatial_scale` in the plugin manifest, and applies either the source override or the default to source/listener relative distance during spatial attenuation, panning, and preview HRTF ear-distance calculation.
- `zircon_plugins/sound/plugin.toml` mirrors the runtime `sound_options()` key set, including `sound.global_volume_gain` and `sound.default_spatial_scale`, so static plugin discovery, Hub catalog views, native/export materialization, and runtime-generated package manifests expose the same profile knobs.
- `playback.rs` now includes `SoundPlaybackStatus`, `SoundPlaybackCompletionAction`, `SoundPlaybackFinished`, and `SoundPlaybackFinishReason`. `SoundPlaybackSettings::{ONCE, LOOP, DESPAWN, REMOVE}` mirror Bevy `PlaybackSettings::{ONCE, LOOP, DESPAWN, REMOVE}` from `dev/bevy/crates/bevy_audio/src/audio.rs`, and the const helper methods cover common initial playback overrides. `SoundManager::pause_playback`, `resume_playback`, `toggle_playback`, `set_playback_gain`, `set_playback_speed`, `seek_playback_seconds`, `mute_playback`, `unmute_playback`, `toggle_mute_playback`, `playback_empty`, `playback_status`, and `drain_finished_playbacks` provide the first Bevy `AudioSinkPlayback`-style lifecycle surface. This intentionally mirrors Bevy's `pause`, `play`, `toggle_playback`, `is_paused`, `set_volume`, `set_speed`, `try_seek`, `mute`, `unmute`, `toggle_mute`, `stop`, `empty`, and playback-position model from `dev/bevy/crates/bevy_audio/src/sinks.rs`. `SoundPlaybackSettings::start_seconds` and `duration_seconds` mirror Bevy `PlaybackSettings::start_position` and `duration` from `dev/bevy/crates/bevy_audio/src/audio.rs` / `audio_output.rs`; `SoundPlaybackCompletionAction::{DespawnEntity, RemoveAudioComponents}` records the intent behind Bevy `PlaybackMode::{Despawn, Remove}` without letting the sound runtime mutate scene ownership directly. The finished queue is the runtime-side bridge for Bevy's `cleanup_finished_audio` path in `dev/bevy/crates/bevy_audio/src/audio_output.rs`.
- `options.rs` defines runtime/editor-facing sound plugin options and quality gates.
- Framework `output.rs` defines the output-device and backend callback contract: descriptors, picker-ready device rows, backend capabilities, start/stop state, callback reports, latency diagnostics, status diagnostics, and render statistics. It is intentionally backend-neutral so the software/null callback path and CPAL/platform driver share the same manager surface without exposing CPAL types.
- Framework `preset.rs` defines serializable mixer preset descriptors. The runtime owns the concrete built-in preset catalog in `zircon_plugins/sound/runtime/src/presets.rs`.
- `zircon_plugins/sound/editor/src/live_output/` defines the editor-facing live output controller and serializable snapshot/action DTOs. It consumes `Arc<dyn SoundManager>` and marks picker rows selected from the current output status without depending on CPAL or concrete runtime internals.

The plugin implementation is split under `zircon_plugins/sound/runtime/src/engine/` so `service_types.rs` remains a manager boundary rather than an audio engine dumping ground. Runtime automation value mapping lives in `zircon_plugins/sound/runtime/src/automation.rs`, sound-owned Timeline sequence scheduling lives in `zircon_plugins/sound/runtime/src/timeline.rs`, shared descriptor validation lives in `zircon_plugins/sound/runtime/src/descriptor_validation.rs`, full graph import lives in `zircon_plugins/sound/runtime/src/mixer_configuration.rs`, dynamic event registry/queue validation lives in `zircon_plugins/sound/runtime/src/dynamic_events.rs`, generic plugin-event ABI projection lives in `zircon_plugins/sound/runtime/src/dynamic_event_abi.rs`, ray-traced impulse-response provider submission lives in `zircon_plugins/sound/runtime/src/ray_tracing.rs`, and concrete output backends live under `zircon_plugins/sound/runtime/src/output/`.

The output subsystem is now folder-backed:

- `output/mod.rs` is the facade that owns neutral descriptor validation, backend/device catalog composition, start/stop/status counters, unavailable-backend recovery state, latency/status diagnostics, and backend session dispatch.
- `output/software.rs` owns deterministic `software-null` and `software-*` capability/support rules plus the always-available software picker row.
- `output/ring_buffer.rs` owns the bounded interleaved `f32` FIFO used by realtime callbacks.
- `output/cpal.rs` owns the optional CPAL adapter behind the `cpal-backend` feature, CPAL device enumeration, picker-session device IDs, CPAL selected-device resolution, and backend queue diagnostics while keeping CPAL types out of the framework DTO crate.

## Behavior Model

The default manager keeps a `SoundEngineState` with clip cache, external source blocks, active clip playbacks, explicit source voices, listener/volume descriptors, automation bindings, parameter values, static impulse responses, mixer graph, DSP runtime state, meters, latency, and ray-tracing status. Clip playbacks and explicit sources keep fractional source cursors so clip and external input blocks can be resampled deterministically into the mixer sample rate.

Audio asset import now has a concrete runtime path instead of only profile-level declarations. `zircon_plugins/audio_importer/runtime/src/lib.rs` registers WAV plus Symphonia-backed MP3/OGG/FLAC/AIFF importers and emits `SoundAsset` entries, while Opus remains declared as a native-backend diagnostic path. `SoundAsset::from_wav_bytes(...)` validates RIFF/WAVE structure, non-zero channel/sample-rate fields, block alignment, sample width, and whole-frame data alignment before producing interleaved `f32` samples. `frame_count()` and `duration_seconds()` provide defensive readback for sound manager clip metadata, matching the Bevy split where `AudioSource` stores bytes and loaders/decoders feed runtime playback (`dev/bevy/crates/bevy_audio/src/audio_source.rs`) while `PlaybackSettings` and sinks consume validated sources (`dev/bevy/crates/bevy_audio/src/audio.rs`, `dev/bevy/crates/bevy_audio/src/sinks.rs`). Importer selection now treats diagnostic-only handlers as lower maturity than available handlers for the same source matcher: a real WAV/codec/runtime importer wins over an externalized warning even when the diagnostic row has a higher priority, and priority remains the tie-breaker when both candidates have the same availability class.

Active clip playbacks now have lifecycle readback plus pause/resume/toggle, mute/unmute/toggle, gain, speed, seek, range control, completion-action intent, non-destructive empty polling, and terminal completion reporting. `playback_status(id)` reports clip ID, loop flag, completion action, paused/muted state, gain, speed, range start/end frames, cursor frame, cursor seconds, and current output track. A paused playback remains registered, renders silence, and does not advance its source cursor until `resume_playback(id)` clears the pause flag; `toggle_playback(id)` flips the same paused flag. A muted playback renders silence while continuing to advance, matching Bevy's managed-mute behavior where volume is suppressed without losing the intended gain; `toggle_mute_playback(id)` flips the same muted flag. `start_seconds` and `duration_seconds` clamp playback to a clip subrange; looped playback wraps within that subrange instead of the whole clip. `seek_playback_seconds(id, seconds)` saturates to the active playback range and updates the same cursor reported by `playback_status`. Natural completion, missing clip removal, and explicit `stop_playback(id)` remove the active playback and append a `SoundPlaybackFinished` record containing the original completion action. `playback_empty(id)` mirrors Bevy `AudioSinkPlayback::empty()`: active playbacks report false, completed or stopped playbacks queued for cleanup report true, and IDs become unknown after `drain_finished_playbacks()` consumes that terminal record. Hosts map `DespawnEntity` / `RemoveAudioComponents` to scene cleanup policy after observing the finished queue.

Spatial source rendering now has Bevy-style default and per-source scale knobs. `default_spatial_scale` defaults to `1.0`, rejects non-finite or negative values, and scales source/listener positions before distance-sensitive spatial calculations. `SoundSpatialSourceSettings::spatial_scale` can override that default for one source, matching Bevy's `PlaybackSettings::spatial_scale` behavior where an individual spatial playback can use project-specific units. This lets a project choose whether one scene unit means one audio meter globally while still allowing exceptional emitters.

Explicit `AudioSource` voices now also carry Bevy `PlaybackSettings::speed`, `PlaybackSettings::muted`, `PlaybackSettings::start_position`, and `PlaybackSettings::duration` style controls from `dev/bevy/crates/bevy_audio/src/audio.rs`. `speed` scales the source cursor for clip-backed sources and must be finite and greater than zero. `muted=true` renders silence while still advancing the cursor, while `playing=false` remains the source-level pause gate and stops cursor advancement. `start_seconds` and `duration_seconds` limit clip-backed source voices to the same deterministic frame-range model used by `play_clip`; looped sources wrap within that range. `source_status(id)` reports the active source input, pause/mute/loop flags, cleanup action, gain/speed, range, cursor, and output track; `source_empty(id)` mirrors the playback empty model by returning false for active sources, true for completed or stopped sources waiting in the finished queue, and unknown after that queue is drained. `pause_source`, `resume_source`, `toggle_source`, `set_source_gain`, `set_source_speed`, `seek_source_seconds`, `mute_source`, `unmute_source`, `toggle_mute_source`, and `stop_source` give scene-owned `AudioSource` voices the same Bevy `AudioSink`-style runtime control surface as clip playbacks, without forcing callers to replace the whole descriptor. Clip-backed sources publish `SoundSourceFinished` records when non-looping playback ends, finite non-looping external source blocks publish the same terminal record when their submitted block is exhausted, and `stop_source` publishes that record for clip, external, synth, or silence inputs with the original input plus optional clip ID. This mirrors Bevy `AudioSinkPlayback::stop` and `cleanup_finished_audio` from `dev/bevy/crates/bevy_audio/src/sinks.rs` and `dev/bevy/crates/bevy_audio/src/audio_output.rs`: the sound runtime removes the finished source voice, records the source ID, input, clip if present, finish reason, output track, and completion action, and lets the host map `DespawnEntity` / `RemoveAudioComponents` to scene cleanup. The controls are exposed through dynamic component metadata, descriptor validation, and the same software mixer path used by `play_clip`; `speed` and `muted` are also available to automation and render-time synth parameter binding.

The `SoundManager` surface now exposes graph mutation at the track/send/effect level instead of requiring callers to replace the whole graph for common authoring operations. Track sends are keyed by target track ID, so adding a send to the same target updates the existing edge; removing a missing edge returns a typed `UnknownSend` error. Synth/automation parameters also support readback through `parameter_value(...)`, with unknown keys returning `UnknownParameter`.

`configure_mixer(...)` is now a full graph import path, not only a track-list replacement. It validates the incoming `SoundMixerGraph`, imports graph-owned `SoundSourceDescriptor` entries into the live source registry with fresh cursors, assigns IDs for authoring sources that omit one, imports graph automation bindings into the runtime binding table, synchronizes the dynamic event catalog, drops stale event handlers, executors, and pending event invocations that no longer match the graph catalog, resets stale DSP/track state, and refreshes meters for the new track set. Duplicate configured source IDs or automation binding IDs return typed `InvalidParameter` errors before state is replaced.

Mixer preset application is now part of the same manager surface. `available_mixer_presets(...)` returns built-in serializable graphs for the default master-only graph, a Music/SFX/Ambience bus layout, and a Spatial Room layout with a room reverb return. `apply_mixer_preset(locator)` validates the selected graph, replaces the active mixer, clears DSP/track runtime state and meters that no longer match the graph, reroutes any active playbacks or sources pointing at removed tracks back to `master`, and removes stale source sends. Direct `remove_track(track)` now applies the same playback/source output-track repair for the removed track, so later playback status and finished events do not point at a deleted mixer bus. Unknown preset locators return `InvalidLocator`.

External component and plugin audio is now represented by `ExternalAudioSourceHandle` plus `SoundExternalSourceBlock`. Other plugin systems can submit a finite interleaved audio block through `submit_external_source_block(...)`; any `AudioSource` using `SoundSourceInput::External(handle)` can render that block into its output track, with the source voice owning playback cursor, loop behavior, and sample-rate conversion into the current mixer rate. `clear_external_source(...)` removes the provider block and returns `UnknownExternalSource` for missing handles.

`AudioSource` parameter bindings are active at render time. Each `SoundSourceParameterBinding` maps a source parameter such as `gain`, `playing`, transform coordinates, or spatial controls to a stable synth parameter ID. The source descriptor stored in the graph is left unchanged; each block resolves a temporary descriptor from the current synth parameter table, so modulation can follow synthesizer output without forcing Timeline or synthesizer crates to own sound internals.

Ray/occlusion now has a first provider-fed runtime path without moving geometry traversal into the sound runtime. `SoundRayTracedImpulseResponseDescriptor` carries optional `occlusion_gain` alongside the existing source/listener/volume cell metadata and IR samples. The sound runtime validates that provider gain as finite `0.0..=1.0`, stores it with the ray-traced IR descriptor, and `engine/occlusion.rs` selects the most specific matching provider row for the active `(source, listener, volume)` query, breaking ties by the descriptor's ray count. When no matching provider row exists, existing `AudioSource.spatial.occlusion_enabled` behavior continues to use the deterministic software fallback gain. This keeps concrete ray-query ownership outside `zircon_runtime::core::framework::sound` and outside the sound plugin renderer while letting provider evidence affect preview output immediately.

Timeline integration now has runtime value, curve, and sequence-entry points. `apply_automation_value(binding, value)` resolves a registered `SoundAutomationBinding`, checks finite input, and applies the scalar to the target track, effect, source, listener, volume, or synthesizer parameter. `apply_automation_curve_sample(binding, curve, time_seconds)` accepts a serializable `SoundAutomationCurve`, validates finite strictly-increasing keyframes, samples step/linear/smooth-step interpolation with endpoint clamping, and then applies the sampled scalar through the same typed binding path. `schedule_timeline_sequence(sequence)` registers or replaces a sound-owned sequence of automation tracks, `advance_timeline_sequences(delta_seconds)` advances all scheduled sequences, samples every bound curve, applies the values through the existing automation target path, returns per-binding sample reports, and removes completed non-looping sequences. Looping sequences wrap by duration. Timeline still owns editor sequence authoring and global playback orchestration, while sound owns deterministic sequence sampling, target resolution, validation, and local scheduling state.

Dynamic events are now an interface-level runtime queue with deterministic multi-plugin fan-out, runtime-local execution, and a stable plugin-event ABI adapter. `dynamic_event_catalog(...)`, `register_dynamic_event(...)`, and `unregister_dynamic_event(...)` manage the sound-owned versioned catalog; unregistering an event also removes its registered handlers, matching runtime executor callbacks, and pending invocations. `register_dynamic_event_handler(...)` stores a plugin-owned handler for a known event, keyed by `(plugin_id, handler_id)` and ordered by `priority` for dispatch. `DefaultSoundManager::register_dynamic_event_executor(...)` attaches an in-process Rust callback to an existing handler descriptor, while `unregister_dynamic_event_executor(...)` removes that callback and reports missing keys through `UnknownDynamicEventHandler`. `DefaultSoundManager::register_dynamic_event_abi_callback(...)` adapts a generic `ZrPluginEventCallbackFnV1` into the same executor path: each `SoundDynamicEventDelivery` is projected into `ZrPluginEventCallbackRequestV1` with the `sound.dynamic_events` namespace, plugin/handler/event IDs, optional source path, event time, payload schema, and opaque payload bytes. Non-OK callback status, non-OK result status, or an unsupported result ABI version becomes a per-handler `Failed` detail without aborting later handlers. `submit_dynamic_event(...)` validates event IDs, finite event time, and payload schema before appending an opaque invocation. `dispatch_dynamic_events(...)` drains pending invocations and returns one `SoundDynamicEventDelivery` per matching handler, sorted by higher priority first and then stable plugin/handler IDs; `execute_dynamic_events(...)` uses the same delivery order, invokes registered executors outside the manager lock, and reports `Succeeded`, `Failed`, or `SkippedMissingExecutor` for every handler without aborting later handlers. `drain_dynamic_events(...)` remains available as a raw queue escape hatch for hosts that want to perform their own dispatch.

Impulse-response cache lifecycle is explicit in the manager surface. `set_impulse_response(...)` installs or replaces a static IR, `remove_impulse_response(...)` invalidates it, and missing removals return `UnknownImpulseResponse`. The ray-traced convolution feature remains gated, but `set_ray_tracing_convolution_status(...)` lets the feature/capability layer publish `WaitingForGeometryProvider`, `StaticImpulseResponse`, or validated `RayTraced` status into mixer snapshots. The optional ray-query path can now call `submit_ray_traced_impulse_response(...)` with a `SoundRayTracedImpulseResponseDescriptor`; the runtime validates source/listener/volume references, optional occlusion gain, format, rays, and finite IR samples, installs the IR into the same convolution cache used by static reverb, and updates `RayTraced { cached_cells, rays_per_update }`. `clear_ray_traced_impulse_response(...)` removes the provider-owned IR and returns the status to `WaitingForGeometryProvider` when the ray cache empties.

Output device lifecycle is now explicit in the manager surface. `available_output_backends(...)` always reports the deterministic `software-null` backend capability used by tests and editor preview, and reports `cpal` only when `zircon_plugin_sound_runtime` is compiled with `--features cpal-backend`. `available_output_devices(...)` returns picker-ready `SoundOutputDeviceInfo` rows with a neutral descriptor, default flag, availability flag, and optional diagnostic. The software row is always available and its descriptor can be passed directly into `configure_output_device(...)`. With `cpal-backend`, CPAL rows are best-effort: `sound.output.cpal.default` represents the current platform default device, and `sound.output.cpal.device.<index>` represents an indexed device from the current enumeration pass. These indexed IDs are picker-session IDs, not persisted OS identifiers. `configure_output_device(...)` validates the backend ID, display name, sample rate, channel count, block size, and latency blocks, then updates the runtime config and mixer snapshot format. Reconfiguration stops the active output backend and clears DSP runtime state because sample-rate/channel changes invalidate delay lines and effect history. If a requested backend is unavailable, including `cpal` without the feature or a host without a usable default output device, the runtime records the requested backend and detail so `backend_name()` reflects the requested backend and `backend_status()` reports `SoundBackendState::Unavailable` with no active backend until a supported backend is configured. While unavailable, `start_output_device(...)`, `render_output_device_block(...)`, and `pull_output_backend_callback(...)` return `BackendUnavailable` instead of starting or pulling from an older descriptor. `output_device_status(...)` now reports an estimated latency from configured block size, latency blocks, and sample rate; CPAL sessions also merge ring-buffer queued/capacity samples when active. This mirrors Godot `AudioDriver::get_output_device_list()`, `get_output_device()`, `set_output_device(...)`, and `get_latency()` in `dev/godot/servers/audio/audio_server.h`, and it mirrors Bevy `AudioOutput::default()` / `audio_output_available(...)` in `dev/bevy/crates/bevy_audio/src/audio_output.rs`, where default device creation failure is diagnostic and audio systems skip playback rather than pretending the device exists.

The CPAL adapter is Windows-first but host-device dependent rather than mandatory. `cpal-backend` adds the optional `cpal = "0.15"` dependency, appends a non-deterministic realtime backend row, and lets `start_output_device(...)` attempt the selected CPAL output device. Manual CPAL descriptors that do not use a picker-session ID fall back to the platform default device to preserve descriptor-driven configuration. Stream configuration currently accepts only `f32` output matching the requested sample rate and channel count; devices that cannot satisfy that contract return structured `BackendUnavailable` detail or unavailable picker diagnostics. The adapter keeps concrete CPAL stream ownership inside a dedicated device thread because CPAL stream internals are not `Send`; the manager stores only the backend session handle, stop flag, producer thread, ring-buffer queue diagnostics, and shared counters.

The realtime callback never locks `SoundEngineState` and never calls `render_mix(...)`. A producer thread renders `descriptor.block_size_frames` through the existing mixer path, pushes interleaved samples into a bounded `SoundOutputRingBuffer`, and backs off when the ring buffer is near capacity. The CPAL callback locks only that ring buffer, drains samples in FIFO order, fills any shortage with `0.0`, and increments callback/underrun counters. Producer render errors are recorded as backend status detail and push a silent block so failure does not become a tight realtime loop. Stopping or reconfiguring the device signals both CPAL threads and drops the active backend session before replacing descriptors.

Spatial rendering now has deterministic HRTF profile loading plus the earlier preview fallback. `load_hrtf_profile(...)`, `remove_hrtf_profile(...)`, and `hrtf_profiles(...)` manage validated neutral profile descriptors with left/right FIR kernels. When an active `AudioListener` references a loaded `hrtf_profile`, the runtime applies the left/right kernels to stereo source audio before the usual source gain/pan step. Loaded profile rendering lives in `engine/hrtf.rs` and stores FIR tail history per `(source, listener, profile)` key, so a source rendered in one-frame or short blocks still produces the delayed kernel contribution in the next `render_mix(...)` call. Missing listener profiles never allocate or reuse loaded-profile state; they continue through the deterministic ear-offset preview path, using listener ear offsets to estimate left/right ear distance, interaural gain, and capped interaural delay. Production SOFA/CIPIC database parsing, interpolation, and optimized binaural convolution remain future work.

DSP runtime state is keyed by `(SoundTrackId, SoundEffectId)` so effect tails belong to the track/effect instance rather than to the descriptor type. Delay, algorithmic reverb, convolution reverb, modulated delay history, modulation phase, phaser phase, compressor release envelopes, and filter biquad histories now keep deterministic state across `render_mix` calls. `engine/filter.rs` computes Audio EQ Cookbook-style biquad coefficients for low-pass, high-pass, band-pass, notch, low-shelf, and high-shelf modes from the existing `SoundFilterEffect` cutoff, resonance, and gain fields, with deterministic finite clamps near Nyquist and per-channel state. Track control delay has its own per-track state, and stale track/effect/HRTF state is pruned when the mixer graph or active source/listener/profile set changes. Mixer snapshots also report a conservative latency frame count derived from enabled delay, reverb, convolution, flanger, chorus, and track delay controls. Production DSP validation now rejects non-finite/out-of-range effect wet mix and core effect parameters before graph insertion, and `add_or_update_effect(...)` revalidates the cloned mixer graph so compressor sidechain track references and post-effect sidechain dependency rules match full graph import.

The render order is a deterministic topological walk over parent routes, track sends, and post-effect sidechain dependencies. This keeps child tracks before parents, send sources before send targets, and post-effect sidechain key tracks before the compressors that read them.

Rendering follows a fixed block pipeline:

1. Validate the graph has a `master` track and no parent cycles.
2. Mix `play_clip` convenience playbacks into their requested output track, resampling clip frames into the configured mixer rate.
3. Mix explicit `SoundSourceDescriptor` voices into dry source buffers, including clip, external block, synth parameter, and silence inputs. Clip and external inputs use the same deterministic linear interpolation source cursor policy.
4. Apply the active listener policy, distance attenuation, stereo pan, source cone, provider-fed ray occlusion or deterministic occlusion fallback, Doppler preview gain, source convolution send, and strongest `AudioVolume` influence to each source buffer.
5. Route post-spatial source buffers into their output tracks, while `SoundSourceSend::pre_spatial` can tap the dry buffer for FMOD-style pre-spatial sends.
6. Process non-master tracks in topological order, preserving both pre-effect and post-effect sidechain tap buffers.
7. Run enabled effect chains, then track controls: gain, pan, L/R trims, delay, phase inversion, mute, solo direct-input gating, and bypass.
8. Route processed tracks into parent tracks and sends.
9. Process `master`, update meters, apply master gain, and clamp final samples.

`play_clip` is retained only as a convenience path into this graph. It is not a separate compatibility mixer.

## Design and Rationale

The contract deliberately models FMOD/Unreal/Godot/Fyrox-style audio structure:

- `master` is mandatory and cannot be removed.
- Custom tracks can route to `master` or another parent and can send into other tracks.
- Explicit send CRUD preserves the graph validator as the single source of truth for missing targets, self-sends, parent cycles, send cycles, and post-effect sidechain dependency cycles.
- Solo tracks mute non-solo direct playbacks/sources while still allowing solo-track audio to route through parent tracks to `master`.
- Effect descriptors are data-only so future native/VM plugin paths can serialize them.
- Sidechain compression references another track by ID instead of holding concrete Rust references. `pre_effects` reads the key track input buffer; post-effect taps require the key track to render earlier and read its processed control-applied buffer.
- Stateful DSP lives beside the mixer graph state, not in the neutral DTO layer. The neutral graph remains serializable while the plugin runtime owns delay lines, history buffers, modulation phase, filter biquad histories, and compressor envelopes that should not be serialized as authoring data.
- Automation writes are binding-driven instead of raw string writes into the graph. Track/effect/source/listener/volume targets expose a first scalar parameter set, unknown bindings return `UnknownAutomationBinding`, unsupported parameter names fail with `InvalidParameter` before mutating state, and curve samples reuse the same mutation path after validating curve data.
- Static convolution reverb and ray-traced provider submissions use the same `SoundImpulseResponseId` cache. Sound owns validation, cache insertion, status snapshots, provider-fed occlusion gain selection, and invalidation; scene geometry traversal and ray queries remain outside the core sound runtime and publish through the provider-facing descriptor.
- Output devices are modeled as backend-neutral DTOs even though the sound plugin now has a concrete CPAL adapter. The `software-null` backend remains the deterministic adapter seam for CI/editor preview, while `cpal` is a feature-gated, host-device-dependent backend that stays entirely inside `zircon_plugins/sound/runtime/src/output/cpal.rs`. Device picker rows expose descriptors and diagnostics, not CPAL handles. CPAL-specific stream/device types do not cross `zircon_runtime::core::framework::sound` or `service_types.rs`.
- External and synthesizer sound sources are neutral handles/parameter IDs, so other plugins can feed audio without linking to sound internals. External sources deliberately submit block data through the manager contract; the sound runtime owns only resampling cursors and routing, not the upstream component. Source parameter bindings reuse the same stable parameter table so synth-driven modulation is local to sound but does not couple to a concrete synthesizer plugin.
- Active listeners are selected deterministically by requested mixer target, then `master`, then stable listener ID. This keeps multi-listener editor previews predictable while leaving room for richer runtime routing later.
- HRTF profile loading is intentionally neutral and deterministic: the runtime accepts explicit profile descriptors and FIR kernels rather than embedding a third-party database parser. It gives authored listener profile IDs real kernel-driven behavior now and stores loaded-profile FIR state in the sound runtime only, while production profile database loading, interpolation, and optimized binaural convolution remain advanced backend concerns.
- `AudioVolume` influence is deterministic in the first runtime slice: the highest-priority volume with non-zero influence wins, and its crossfade blends exterior/interior gain, low-pass, and static convolution send weight.

The first CPU DSP implementations are deterministic block effects for tests and editor preview. Delay, algorithmic reverb, convolution, modulated delay, phaser LFO phase, compressor release, stateful biquad filters, loaded HRTF FIR tails, and track delay now preserve history across render calls, but the DSP path is still a first-stage software renderer rather than a final low-latency audio-device backend.

## Edge Cases and Constraints

- Removing `master` is rejected.
- Parent cycles, send cycles, and post-effect sidechain cycles are rejected before replacing the stored graph. Effect updates use the same graph validator, so missing sidechain tracks and same-track post-effect sidechain taps are rejected before mutating the live graph.
- Missing track references in playbacks, sends, or parent routes return typed errors.
- Playback settings reject non-finite initial gain, pan, speed, start time, and duration before inserting a live playback. Playback speed must be positive, and zero-duration slices are rejected because they cannot cover a source frame.
- Mixer preset application rejects unknown locators and cleans up live playbacks/source routes that point at tracks removed by the selected preset. Direct track removal reroutes live playback/source outputs from the removed track to `master`.
- Clip and external source resampling uses deterministic linear interpolation. Non-looping sources clamp interpolation at the final frame before reporting end-of-source, while looping sources interpolate across the wrap point. Active playback subranges report completion at the range end, and looped subranges interpolate/wrap back to the configured range start.
- External source blocks reject empty provider handles, zero sample rate, zero channels, non-finite samples, and partial frames; clearing a missing external handle returns `UnknownExternalSource`.
- Source parameter bindings reject empty IDs and unsupported source parameter names before insertion. Synth parameter inputs reject empty parameter IDs and non-finite default values. Missing synth parameter values are treated as absent modulation for that block rather than as render failures.
- Output device descriptors reject empty IDs/backend/display names, zero sample rates, zero channel counts, zero block sizes, zero latency blocks, and unavailable backend IDs. Unavailable backend IDs are retained in `backend_name()` / `backend_status()` as structured `Unavailable` diagnostics until a supported backend is configured; while this state is active, starting or pulling output returns `BackendUnavailable` with the unavailable detail. Pulling output or backend callbacks before start returns `BackendUnavailable` with a stopped-device detail. Selecting `cpal` without compiling `cpal-backend` produces a feature-disabled unavailable diagnostic and can recover by reconfiguring `software-null`. With the feature compiled, device enumeration may still return unavailable CPAL rows when the host has no default device, cannot enumerate devices, or cannot satisfy the runtime's current `f32` sample-rate/channel contract. Startup can still return unavailable if the selected/default device disappears, rejects the requested format, or denies stream creation/playback.
- Removing a missing impulse response returns `UnknownImpulseResponse`; ray-traced status rejects `RayTraced` snapshots with zero rays per update. Provider-submitted ray-traced IRs reject empty cell keys, zero sample rates/channels, zero ray counts, non-finite samples, occlusion gains outside `0.0..=1.0`, and unknown source/listener/volume references.
- Dynamic event descriptors reject empty IDs, display names, or payload schemas; handler descriptors reject empty plugin IDs, handler IDs, event IDs, or display names and must reference a registered event. Submitted invocations reject unknown events, mismatched schemas, and non-finite event times. Runtime executor and ABI callback registration require an existing handler descriptor, duplicate registration replaces the callback, and missing executor removal returns `UnknownDynamicEventHandler`. ABI callbacks that return a failing `ZrStatus`, a failing `ZrPluginEventCallbackResultV1`, or the wrong result ABI version are reported as handler failures instead of aborting the batch. Unregistering an event also removes registered handlers, matching executors, and pending invocations for that event ID; removing a missing handler returns `UnknownDynamicEventHandler`.
- Removing a missing track send returns `UnknownSend`; querying a missing sound parameter returns `UnknownParameter`; applying or removing a missing automation binding returns `UnknownAutomationBinding`.
- Effect wet mix must stay finite within `0..=1`. Core DSP parameters are validated before graph insertion: gain and trim scalars must be finite; filter cutoff/resonance/gain must be finite with positive cutoff and non-negative resonance; reverb room size/damping and phaser depth must stay within `0..=1`; compressor ratio must be finite and `>= 1` with non-negative attack/release; delay/flanger/phaser feedback must stay finite within `0..=0.99`; shaper drive and modulation rates must be non-negative; stereo pan must be finite within `-1..=1`; and limiter ceiling must be finite and positive.
- Source descriptors reject non-finite position/forward/velocity/gain values and spatial ranges outside the supported envelope.
- Listener descriptors reject non-finite transforms and unknown mixer targets.
- HRTF profiles reject empty IDs/display names, zero sample rates, empty kernels, non-finite kernel samples, and all-zero kernels. Removing a missing profile returns `UnknownHrtfProfile`. Rendering with a missing listener profile falls back to preview instead of failing, and replacing/removing profiles or reconfiguring the output/mixer clears loaded HRTF runtime history.
- Volume descriptors reject invalid shapes, non-finite gains, negative crossfade distances, and invalid low-pass cutoffs.
- Impulse responses must be non-empty and finite.
- Automation bindings must name a Timeline-style track path. Runtime value application, curve sampling, and sound-owned Timeline sequence scheduling are available for scalar targets. Timeline sequence descriptors reject empty IDs, non-positive or non-finite durations, empty track lists, duplicate bindings, missing automation bindings, invalid curves, and keyframes outside the sequence duration.
- Ray-traced convolution does not trace geometry itself until a geometry/ray-query provider exists; the provider submission path accepts computed IRs, feeds the existing convolution cache, and can publish an occlusion gain for the matching source/listener/volume cell.
- CPAL output now includes neutral device enumeration, picker-session descriptors, and a plugin-local editor snapshot/action model, but not retained-host data binding, persisted OS device IDs, per-host latency tuning, cross-platform manual acceptance beyond the Windows-first smoke path, lock-free production scheduling, sample-format conversion, or input/capture support. Those stay future backend work rather than weakening the deterministic software output path.
- The editor live output model is intentionally a plugin-local controller over `SoundManager`, not a `zircon_editor` pane-payload specialization. Generic host binding can consume the snapshot later without adding sound-specific branches to editor core.

## Plan Sources

This update implements the first "core first" stage from `Sound 插件核心完善计划`: stable contracts, graph/DSP runtime, components, options/dependencies, dynamic event registry/queue/execution contracts, graph-owned source/automation import, ray-traced IR provider submission, and editor authoring descriptors.

The DSP/HRTF quality slice follows `docs/superpowers/specs/2026-05-20-sound-dsp-hrtf-quality-design.md` and `docs/superpowers/plans/2026-05-20-sound-dsp-hrtf-quality.md` by keeping new runtime-only state in `zircon_plugins/sound/runtime/src/engine/filter.rs`, `engine/hrtf.rs`, `engine/dsp_state.rs`, and `engine/state.rs` while leaving neutral sound DTOs unchanged.

The CPAL adapter slice follows `docs/superpowers/specs/2026-05-21-sound-cpal-adapter-design.md` and `docs/superpowers/plans/2026-05-21-sound-cpal-adapter.md` by keeping the framework output DTOs neutral, moving backend implementation under `zircon_plugins/sound/runtime/src/output/`, adding `cpal-backend` as an optional dependency gate, and preserving `software-null` as the deterministic acceptance backend.

The CPAL polish slice follows `docs/superpowers/specs/2026-05-23-sound-cpal-polish-design.md` and `docs/superpowers/plans/2026-05-23-sound-cpal-polish.md` by adding `SoundOutputDeviceInfo`, `SoundOutputLatencyStatus`, `SoundManager::available_output_devices(...)`, deterministic software picker rows, feature-gated CPAL picker rows, picker-session CPAL device IDs, and status latency/queue diagnostics while keeping concrete CPAL handles inside `zircon_plugins/sound/runtime/src/output/cpal.rs`.

The editor live output slice follows `docs/superpowers/specs/2026-05-23-sound-editor-live-output-design.md` and `docs/superpowers/plans/2026-05-23-sound-editor-live-output.md` by keeping live output projection inside `zircon_plugins/sound/editor/src/live_output/`, consuming only `SoundManager` and neutral DTOs, and exposing mixer-toolbar control metadata without adding a sound-specific branch to `zircon_editor` core.

The dynamic event ABI follow-up extends `docs/superpowers/specs/2026-05-04-sound-dynamic-event-execution-design.md` and `docs/superpowers/plans/2026-05-04-sound-dynamic-event-execution.md` without moving callback ownership into the neutral sound framework. `zircon_runtime_interface` owns only the generic `ZrPluginEvent*` ABI structs and the optional `ZrPluginApiV1::invoke_event` slot, while `zircon_plugins/sound/runtime/src/dynamic_event_abi.rs` owns the sound-specific projection into that ABI and reuses the existing deterministic executor path.

The deeper follow-up remains:

- generic retained-host binding of the live output snapshot into picker rows plus persisted device preference policy for the CPAL output adapter,
- per-host latency tuning, cross-platform manual device validation, sample-format conversion, lock-free callback scheduling, and input/capture support,
- production-grade low-latency DSP details such as denormal handling, SIMD/partition tuning, high-order resampler tuning, and audio-device scheduling,
- production HRTF database parsing/interpolation/optimized binaural convolution and geometry-backed occlusion,
- actual ray traversal/impulse-response generation from scene geometry,
- generic native-dynamic plugin loader wiring that discovers `ZrPluginApiV1::invoke_event` callbacks and attaches them to registered subsystem handler descriptors,
- host-side editor operation handlers that dispatch registered sound operations through generic plugin command routing instead of sound-specific editor-core branches.
