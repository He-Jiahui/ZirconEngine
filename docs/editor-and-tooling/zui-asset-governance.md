# ZUI Asset Governance

`.zui` is the production suffix for reusable UI component prototypes. A `.zui` document uses the UI v2 TOML schema in memory, but it is not a view root. `UiZuiAssetLoader` enforces the component profile: `asset.kind = "component"`, no `[root]` view entry, exactly one component, and a component root node that exists in the node table.

Production `.zui` files must live under explicit component directories such as `ui/editor/components` or `ui/editor/material_components`. Files outside those directories are allowed only as registered builtin alias exceptions; the current host-path exception is `ui/editor/host/activity_drawer_window.zui`. This keeps reusable component prototypes from being scattered through view/root folders while still allowing a short migration window for stable builtin ids.

Production `.v2.ui.toml` files remain the builtin view/style roots. A production `.v2.ui.toml` asset may declare only `asset.kind = "view"` or `asset.kind = "style"`; component prototypes belong in `.zui` files. Their `imports.widgets` entries may reference only `.zui` component assets or explicitly registered builtin `.zui` aliases. A direct widget import must name the component fragment explicitly, for example `res://ui/editor/components/showcase_visual_section.zui#ShowcaseVisualSection`; alias imports follow the same `asset.id#ComponentName` shape. Importing a `.zui` file without `#ComponentName`, or importing an old `.v2.ui.toml#ComponentName` widget source, is invalid for production assets because widget composition now flows through `.zui` component prototypes.

Production `.zui` `[asset].id` values default to their file-derived `res://` locator. This keeps component assets relocatable through the same path alias that `UiV2PrototypeStoreFileCache` registers for loaded source files. Builtin aliases are allowed only when they are explicitly documented in the governance test and are still referenced by a production `.v2.ui.toml` root. The current exception is `zircon_editor/assets/ui/editor/host/activity_drawer_window.zui`, whose stable id is `editor.host.activity_drawer_window`; the window roots import that alias while the file cache also exposes the path alias `res://ui/editor/host/activity_drawer_window.zui`.

Production UI asset ids are globally unique across `.v2.ui.toml` view/style roots and `.zui` component prototypes. A `.zui` component may use a registered builtin alias instead of its locator during migration, but that alias still participates in the same production asset id namespace and must not collide with a root view, style asset, or another component.

Production UI asset headers must stay authorable and current. Every production `.v2.ui.toml` and `.zui` asset must declare the current UI v2 schema version plus non-empty, trimmed `asset.id` and `display_name` fields, so the Widget Editor, asset browser, diagnostics, and future migration tools can present stable identifiers and human-facing labels instead of deriving names from paths or carrying invisible whitespace differences.

Production `.zui` component names must stay derived from their file stems. The default component name is the file stem converted from snake case to PascalCase, such as `showcase_visual_section.zui` declaring `ShowcaseVisualSection`. Authoring/demo prototype assets may append `Prototype`, such as `material_alert.zui` declaring `MaterialAlertPrototype`. This gives widget imports, palettes, previews, and migration tools a predictable relationship between file paths and component fragments.

Production `.zui` component names must also be clean authoring tokens: non-empty, trimmed, and whitespace-free. Component names become import fragments, palette entries, prototype-store keys, and diagnostics labels; allowing whitespace variants would make the same public component contract appear under multiple spellings.

Production `.zui` component roots must resolve to concrete authoring nodes, not `Slot` placeholders. Slots are projection points inside a reusable component; making one the component root would collapse the component contract into unowned projected content and leave previews, hierarchy views, and instance expansion without a stable root surface.

Production `.zui` component root nodes must declare explicit layout metadata, and that layout must include a `width` contract with a clean `stretch` mode. The root node is the mount surface used by previews, hierarchy editing, slot projection, and runtime surface composition; requiring an explicit horizontal sizing contract at that entry point prevents components from depending on implicit host defaults that the Widget Editor cannot display or edit reliably.

Production `.zui` components that declare `default_classes` must expose at least one usable class token. The class list is allowed to be absent for wrapper-only components, but once present it becomes the public style anchor used by default theming, previews, palettes, and Widget Editor class forms; an effectively empty list would look intentional in source while carrying no selectable styling contract.

Every production `.zui` file must be reachable from at least one production `.v2.ui.toml` widget import. Reachability can be direct, through `res://...zui#ComponentName`, or through an explicitly registered builtin alias such as `editor.host.activity_drawer_window#ActivityDrawerWindow`. This prevents component prototypes from drifting as orphan files after showcase or workbench roots move.

Inside each production `.zui` component asset, the node table must also be fully reachable from the single component root through `children` links. `.zui` is a component prototype, not a scratch document, so unreachable node definitions are treated as stale asset data that would confuse previews, diffs, migration tools, and future Widget Editor hierarchy views.

Production `.zui` component root node references must be non-empty and trimmed. The root is the component prototype entry point used by previews, hierarchy traversal, slot projection, and future Widget Editor selection state; invisible empty or whitespace-padded root ids would make those surfaces disagree before child links are even evaluated.

Production `.zui` node ids must stay lower_snake_case. Node ids are persistent authoring identifiers used by hierarchy diffs, child links, preview selection, and future Widget Editor edit state; keeping them lowercase and path-like avoids accidental divergence from display labels or control ids.

Production `.zui` child mount node references must be non-empty and trimmed. Child mounts are the serialized hierarchy edges between persistent node ids; allowing empty strings or whitespace-padded references would make source diffs, slot editing, preview selection, and hierarchy diagnostics disagree about the actual parent-child link.

Production `.zui` node attribute keys must be non-empty and trimmed wherever author-facing node metadata is declared. This applies recursively to `props`, `state`, `layout`, `style.self`, and `style.slot` tables. Attribute keys flow into runtime metadata, style resolution, layout inference, retained-host inspection, and Widget Editor forms; whitespace-only key differences would make those surfaces disagree about which field is being edited.

Production `.zui` node attribute strings that look like file-backed resources must stay portable. Resource-like strings in `props`, `state`, `layout`, `style.self`, or `style.slot` may use logical asset paths such as `res://...` or forward-slash asset-relative paths, but they must not use `dev/` source-tree paths, `../` parent-relative paths, drive-letter absolute paths, or backslashes. This keeps component prototypes relocatable between editor/runtime asset roots, packaged builds, hot reload, and future Widget Editor resource pickers.

Production `.zui` layout metadata must stay inside the shared layout vocabulary. Layout tables may use only the governed top-level keys `width`, `height`, `container`, and `clip`. Axis tables under `layout.width` and `layout.height` may use only `stretch`, `min`, `preferred`, and `max`; container tables may use only `kind`, `gap`, `columns`, and `sequential`. This prevents source assets from carrying ad-hoc layout keys that parse differently in the runtime layout pass, the Widget Editor Inspector, and future visual resize handles.

Production `.zui` axis layout metadata must keep width/height sizing semantics explicit. When an axis declares `stretch`, that value must be a clean string and one of the runtime-supported modes: `Fixed` or `Stretch`. `Stretch` axes must not also carry explicit `min`, `preferred`, or `max` size constraints, because those constraints make the authored axis mix fill semantics with fixed sizing semantics. A `Fixed` axis must declare an explicit `preferred` size; fixed sizing without a preferred bound is ambiguous for previews and authoring resize handles. Numeric axis constraints must remain finite, non-negative, and ordered as `min <= preferred <= max` whenever those fields are present.

Production `.zui` container layout metadata must also stay inside the shared authoring vocabulary. When `layout.container` is present, it must be a table with a clean `kind` string. The current production component profile allows the container kinds already exercised by production assets and runtime layout support: `HorizontalBox`, `VerticalBox`, and `Masonry`. Optional `gap` spacing must be numeric, finite, and non-negative. `Masonry` containers must declare a positive integer `columns` value, and an optional `sequential` value must be boolean. This keeps the source container contract aligned with runtime measure/arrange behavior and prevents the Widget Editor from presenting unknown container controls or invalid spacing/grid controls.

Production `.zui` clip metadata must be boolean when it is present under `layout.clip`. Clip changes affect visual overflow, preview bounds, and hit/selection overlays; keeping the value typed as a real boolean prevents the runtime surface, retained-host projection, and Widget Editor Inspector from disagreeing about whether clipping is enabled.

Production `.zui` node component type strings must be non-empty and trimmed. These strings are the bridge from authored hierarchy nodes to native controls or imported component prototypes; invisible empty values or whitespace-only differences would make prototype expansion, diagnostics, and future Widget Editor palettes harder to reason about.

Production `.zui` node component type strings must also resolve to a known component source. A node may reference a registered editor showcase or Material foundation descriptor id, the current document's local component name, or a component fragment imported through `.zui` `imports.widgets`. Inline `asset#Component` locators inside `node.component` are invalid; asset references belong in the import list so the prototype store, dependency graph, hot reload, and Widget Editor palette all see the same component boundary.

Each production `.zui` node graph must remain a single-root tree. The component root cannot be mounted as a child, and any other node may be mounted by only one parent. This keeps component instancing deterministic and gives the Widget Editor a stable hierarchy instead of a graph with aliases, cycles, or multiple ownership paths.

Within a single `.zui` component asset, `control_id` values must be non-empty, trimmed, whitespace-free, selector-safe, and unique. The allowed character shape starts with an ASCII letter or digit and then uses only ASCII letters, digits, `_`, or `-`. The same `control_id` may appear in another asset, but duplicates, whitespace-only differences, multi-token ids, or punctuation-heavy ids inside one component would make preview selection, binding diagnostics, retained-host testing, and Widget Editor hierarchy tooling ambiguous.

Production `.zui` child mount slot metadata must stay inside the named-slot vocabulary. A child mount `slot = { ... }` map may currently declare only `name` or the migration-compatible alias `slot_name`; parent-container layout fields such as width, height, alignment, padding, rows, columns, or z-order are not valid in this map until the shared PanelSlot contract is explicitly governed. Production `.zui` slot metadata must also use non-empty, trimmed names and keys. This applies to component-declared slot names, node slot maps, child mount slot keys, and child mount `slot.name` values. The rule deliberately does not enforce a casing convention, because imported Material/MUI semantics include camelCase names such as `inputRoot` and `popupIndicator`; it only removes invisible empty or whitespace-padded values that would destabilize hierarchy editing and Inspector state.

Production `.zui` slot name aliases must stay consistent when both spellings appear. The loader accepts `name` and `slot_name` on `Slot` placeholder props and child mount `slot` metadata for migration compatibility, but a source entry that declares both aliases must declare them as strings with the same value. Authors should normally keep a single spelling in production assets; double-written aliases are valid only when the migration path needs them, and they cannot describe two different projection targets.

Production `.zui` `Slot` placeholder nodes must name a component-declared slot through `props.name` or `props.slot_name`. Missing, non-string, whitespace-padded, whitespace-containing, or undeclared placeholder names are invalid because slot projection is the authored boundary between a reusable component prototype and the children mounted by importing views. Every component-declared slot must also have at least one `Slot` placeholder node, so public slot schemas cannot drift into unprojected authoring metadata.

Production `.zui` child mounts that target a known component slot schema must use a declared parent slot name. The known schema set is built from the editor showcase registry, the Material editor foundation registry, and the current `.zui` component's own declared slots. Unknown component ids remain format-checked but are not treated as schema violations until their descriptors are registered, which lets generated Material prototypes and local authoring assets converge without hard-coding temporary slot lists.

Production `.zui` child mounts that fill known component slot schemas must also respect slot cardinality metadata. A non-`multiple` slot may receive at most one child. When a node authors children into any known slot on a component whose schema declares required slots, those required slots must be present as well. This aligns source assets with the same `required` and `multiple` component contract that runtime instancing and the future Widget Editor hierarchy/slot overlay need to enforce.

Production `.zui` event bindings must stay authorable and dispatchable. Each binding id must be non-empty, trimmed, whitespace-free, unique inside its `.zui` asset, globally unique across production `.zui` component assets, and use the slash-delimited authoring path form with at least two ASCII identifier segments, such as `UiComponentShowcase/ButtonCommit` or `MaterialLab/Tabs/Change`. Binding event kinds must stay inside the governed production set currently exercised by editor assets: `Change`, `Click`, `DragBegin`, `DragEnd`, `DragUpdate`, `Drop`, `Hover`, `Press`, `Scroll`, `Submit`, and `Toggle`. A binding may dispatch through a legacy route or a structured action target, but it must declare at least one of them; when a route is present, the route string must be non-empty, trimmed, whitespace-free, globally unique across production `.zui` component assets, and use the dotted authoring route form with at least two ASCII identifier segments, such as `UiComponentShowcase.Commit.Button`. Routed bindings must also keep the leading slash-path id namespace aligned with the leading dotted route namespace, for example `MaterialLab/...` with `MaterialLab...`; this keeps Binding Inspector diagnostics, preview interaction, deterministic replay, route tables, and future action-target migration from carrying duplicate, inert, cross-namespace, or host-unsupported bindings, and lets replay traces or diagnostics identify one authored binding without asset-local disambiguation. New event kinds such as focus, blur, resize, release, or double-click must first extend the shared dispatch/projection contract and this governance rule before appearing in production `.zui` assets.

Production `.zui` component prototypes must keep `style_scope = "closed"`. The shared UI plan allows open style scopes only when a component explicitly exposes public style parts; that contract is not present in the current production `.zui` profile. Keeping component styles closed prevents parent view roots from accidentally depending on private internal node ids, classes, or selector structure before the Widget Editor can expose and validate public style parts.

Production `.zui` class lists must be clean where they are declared. Component `default_classes` and node `classes` may be absent, but present lists must not contain empty, whitespace-padded, whitespace-containing, selector-unsafe, or duplicate class tokens. Zircon-owned class tokens start with a lowercase ASCII letter, may contain lowercase ASCII letters, digits, `_`, or `-`, and must end with a lowercase ASCII letter or digit. Imported Material/MUI compatibility tokens may keep their upstream `MuiName-part` casing, such as `MuiAlert-root` or `MuiChip-deleteIcon`. This keeps selector matching, style diagnostics, preview overlays, and future class editing from carrying redundant, invisible, case-drifting, or punctuation-heavy tokens while preserving recognizable upstream style hooks.

Production `.zui` Material/MUI compatibility style anchors must follow the same selector-safe token quality. Node `props.className`, `props.baseClassName`, and nested `props.slotProps.*.className` values may carry one or more whitespace-separated class tokens, but the prop value must be a non-empty, trimmed string and each token must pass the same class token rule used for component `default_classes` and node `classes`. This lets generated Material prototypes keep upstream prop names while still giving selector matching, Style Inspector, and future class editing deterministic anchors.

Production `.zui` Material/MUI compatibility `slotProps` maps must target known component slots when the node's component descriptor is registered. The rule checks components from the editor showcase and Material foundation registries: `props.slotProps` must be a table, and each key under it must name a slot declared by the node component's descriptor. Registry descriptors must expose compatibility public parts such as `ChatComposer.root` and the Timeline `separator`, `connector`, and `dot` parts before production assets can style them through `slotProps`. Unknown component descriptors remain outside this rule until they are registered. This keeps compatibility props aligned with the same slot schema used by child mounts, Style Inspector parts, and future public-part editing.

Production `.zui` node `props` must remain stable authoring metadata. Top-level and nested prop keys may use Zircon snake_case or upstream Material/MUI camelCase, but they must still be non-empty ASCII identifiers made from letters, digits, and `_`, starting with a letter or `_`. Prop values must stay inside Inspector-serializable TOML primitives, arrays, and tables: strings may be intentionally empty but must be trim-stable when present, numbers must be finite, and TOML datetimes are encoded as strings instead of native datetime values. This keeps Inspector field addressing, source round-tripping, host projection, and generated Material compatibility props from depending on punctuation-heavy keys, invisible whitespace, or value types the UI property editor cannot faithfully display.

Production `.zui` inline style overrides must stay inside the currently governed runtime-recognized style vocabulary. `style.self` may use the render/text keys already exercised by production assets and runtime style dirty classification, such as `background`, `foreground`, `border`, `opacity`, `button_variant`, `button_color`, `button_size`, `button_interaction_state`, and `icon_placement`. `background`, `foreground`, and `border.color` must use clean `material.*` theme tokens, while `border.width`, `border.radius`, and `opacity` must be numeric, finite, and non-negative. This prevents ad-hoc style fields, raw colors, or invalid numeric paint data from bypassing Style Inspector, theme hot reload, and runtime render/style invalidation rules.

If a `.zui` component asset declares its own imports, those imports follow the same boundary as production roots. Component/widget imports must target `.zui#ComponentName` or a registered builtin `.zui` alias with a fragment, and the fragment must name the component declared by the target `.zui` asset. Style imports must be fragment-free `res://` locators that resolve to style assets. This keeps future component nesting and component-local styling on the same asset graph instead of creating a second import dialect.

Production `.zui` widget imports must not point back at the same component asset. The rule checks both the file-derived `res://...zui` locator and any registered builtin alias for that file, so future component expansion cannot create a trivial self-recursive prototype through either import spelling.

Production UI import lists must not repeat dependencies within the same asset. This applies to both `.v2.ui.toml` and `.zui` `imports.widgets` / `imports.styles` lists, keeping dependency graphs deterministic for hot reload, diagnostics, cache invalidation, and future authoring tools.

Production UI import entries must be non-empty and trimmed before dependency resolution. This applies to both `.v2.ui.toml` and `.zui` `imports.widgets` / `imports.styles` values, so dependency graphs, hot reload keys, diagnostics, and Widget Editor import forms do not disagree because of invisible whitespace around a locator or component fragment.

Production `.v2.ui.toml` `imports.styles` entries must use `res://` locators and resolve to UI v2 documents whose `[asset].kind` is `style`. Style imports do not name component fragments; fragment-bearing imports are reserved for widget/component prototypes. This keeps theme chaining explicit and prevents a view root from silently importing another view or component document through the style pipeline.

The builtin template registry must not directly register `.zui` files. Registry entries point at `.v2.ui.toml` roots such as `component_showcase.v2.ui.toml`; those roots then import `.zui` prototypes through the v2 prototype store. This keeps component prototypes reusable without making them standalone windows.

`zui_asset_governance.rs` guards this boundary across production editor/runtime UI roots, with `zui_asset_governance/support.rs` owning shared asset-root scanning, `res://` resolution, widget import parsing, file-stem component naming, and registered builtin `.zui` alias lookup. The rule file parses each `.v2.ui.toml`, requires the document kind to stay `view` or `style`, requires every widget import to target a direct `.zui` locator or registered builtin `.zui` alias, requires an explicit `#ComponentName`, resolves the import to a production `.zui` source file, parses the `.zui` document, and verifies the fragment names the declared component. It also checks that production `.zui` files live under component directories unless their path is an explicit alias exception, requires each `.zui` component name to match the file stem PascalCase with an optional `Prototype` suffix, checks that every production `.zui` source is reachable from widget imports, applies the same widget/style import rules to imports declared inside `.zui` component assets, verifies `.zui` internal widget import fragments against the target component document, rejects `.zui` widget imports that self-reference the current component asset, requires import entries to be non-empty and trimmed, rejects duplicate widget/style import entries inside each production UI asset, verifies every production style import resolves to a style document, scans production `.zui` files to require `[asset].id` to match the derived locator unless the file is listed as a builtin alias and that alias is actively referenced, requires production UI asset ids to stay unique across `.v2.ui.toml` roots and `.zui` components, requires production UI asset headers to use the current schema version and non-empty trimmed asset id/display name fields, and confirms `builtin_template_documents()` contains no `.zui` paths.

`zui_asset_governance/component.rs` owns component-level metadata: component names must be clean authoring tokens, present `default_classes` lists must expose at least one usable public style anchor, and component style scopes must remain closed until public style parts are governed. `zui_asset_governance/component_root.rs` owns component-root contracts: component roots must resolve to concrete non-`Slot` authoring nodes, root nodes must declare explicit layout metadata, root layout must include a `width` contract, and the root width contract must declare a clean `stretch` mode. `zui_asset_governance/graph.rs` owns the intra-component graph invariants: walking each `.zui` component root to ensure the full node table is reachable, requiring component root node references to stay non-empty and trimmed, requiring persistent node ids to stay lower_snake_case, requiring child mount node references to stay non-empty and trimmed, and rejecting root-as-child and duplicate-parent node mounts so each component graph remains a single-root tree. `zui_asset_governance/node_metadata.rs` owns generic node metadata governance: node `props`, `state`, `layout`, `style.self`, and `style.slot` keys must stay non-empty and trimmed, and resource-like strings inside those metadata maps must use portable asset paths instead of `dev/` paths, parent-relative paths, drive-letter absolute paths, or backslashes. `zui_asset_governance/node_component.rs` owns node component reference governance: authored `node.component` strings must stay non-empty and trimmed, avoid inline `asset#Component` locators, and resolve through local component names, `.zui` widget import fragments, or registered editor showcase / Material foundation descriptor ids. `zui_asset_governance/control.rs` owns persistent control identifier governance: optional node `control_id` values must stay non-empty, trimmed, whitespace-free, selector-safe, and unique inside each `.zui` asset so previews, binding diagnostics, retained-host tests, and hierarchy selection resolve one authored control. `zui_asset_governance/class.rs` owns style-anchor governance: component `default_classes`, node `classes`, Material/MUI-compatible `className` and `baseClassName`, and nested `slotProps.*.className` values must stay selector-safe, non-empty when present, trim-stable, and duplicate-free where a list is authored. `zui_asset_governance/event.rs` owns event-binding governance: governed event kind vocabulary, slash-delimited binding ids, asset-local and global id uniqueness, dotted legacy routes, route namespace alignment with ids, route uniqueness, and route-or-action dispatch target coverage. `zui_asset_governance/props.rs` owns generic node prop authoring safety: ASCII identifier keys across nested prop tables, trim-stable strings, finite numeric values, and string-encoded date/time data for Inspector serialization. `zui_asset_governance/style.rs` owns inline style override governance: known `style.self` keys, `material.*` color-token tables, and non-negative numeric render values for `border` and `opacity`. `zui_asset_governance/layout.rs` owns shared layout vocabulary and container/clip constraints: known layout, axis, and container keys, known container `kind` values, non-negative numeric container gaps, Masonry-specific `columns` / `sequential` contracts, and boolean `clip` metadata. `zui_asset_governance/layout_axis.rs` owns width/height axis constraints: known `stretch` modes, unambiguous `Stretch` versus `Fixed` semantics, explicit preferred size contracts for `Fixed` axes, and finite, non-negative, ordered numeric `min` / `preferred` / `max` ranges. `zui_asset_governance/slot.rs` owns generic slot metadata boundaries: keeping child mount slot metadata inside the named-slot key vocabulary, keeping slot metadata names/keys non-empty and trimmed, keeping `name` / `slot_name` migration aliases consistent when both appear, and requiring `Slot` placeholder nodes and declared component slots to stay bidirectional with whitespace-free names. `zui_asset_governance/slot_schema.rs` owns schema-bound slot governance:
