# UI v2 Foundation

UI v2 moves authored UI assets away from recursive runtime `UiTemplateNode` documents and into flat arena data. The stable DTOs live in `zircon_runtime_interface::ui::v2`; runtime loading, heap prototype storage, arena compilation, style resolution, and surface construction live in `zircon_runtime::ui::v2`.

## Runtime Contract

`UiV2AssetDocument` has three top-level asset kinds: `view`, `component`, and `style`. Nodes are stored in a flat `nodes` table and use fixed fields: `component`, `classes`, `props`, `state`, `layout`, `slots`, `events`, and `children`. V2 style declarations are owned by `UiV2StyleSheet`, `UiV2StyleRule`, and `UiV2StyleDeclarationBlock` instead of reusing the legacy template asset style DTOs, while bindings and import lists still reuse existing shared UI contract types.

The component asset importer boundary now reserves `.zui` as the production UI component suffix while still decoding TOML into the existing `UiV2AssetDocument` DTO. `UiZuiAssetLoader::load_zui_str` and `load_zui_file` first parse through the v2 TOML loader, then enforce the `.zui` profile: `asset.kind = "component"`, no top-level `[root]` view entry, exactly one `[components.*]` definition, a non-empty component root, and a component root that resolves into `nodes`. Production `AssetImporter::default()` now routes `.zui` through `import_ui_zui_asset` and materializes only `UiV2ComponentAsset` / `ResourceKind::UiWidget`; `.v2.ui.toml` and `.ui.toml` importers are rejected outside explicit migration fixtures. The split `ui_document_importer` plugin mirrors that production path with `ui_document_importer.zui_component`, `full_suffixes = [".zui"]`, and no view/style output kinds.

Editor extension registration enforces the same boundary for authoring descriptors: `EditorUiTemplateDescriptor` and `ComponentDrawerDescriptor` UI document paths must point to `.zui`. This catches stale plugin registrations before workbench projection tries to load a missing prototype and keeps component drawer metadata on the v2-backed cache/surface path.

Runtime plugin UI component descriptors now enforce the same document boundary. `RuntimeExtensionRegistry::register_ui_component(...)` still validates the plugin-owned component id prefix, then rejects any `ui_document` that does not end in `.zui`. The runtime registry projection keeps the `plugin-ui-component` descriptor role and `ui_document` prop, but stale recursive `.ui.toml` and broad `.v2.ui.toml` documents cannot enter the runtime component catalog through plugin manifests.

The v2 payloads intentionally keep the existing resource categories for now: legacy v2 `view` maps to `ResourceKind::UiLayout`, `component` maps to `ResourceKind::UiWidget`, and `style` maps to `ResourceKind::UiStyle`. New `.zui` project scans use only the component branch and restore artifacts as `ImportedAsset::UiV2Component`, so component prototypes keep the v2 document payload without reopening the old view/style production importer.

`UiV2DocumentCompiler` validates missing references, empty components, multiple parents, and cycles with an explicit stack before building `UiV2NodeArena`. The compiled artifact is now the interface-owned `UiV2CompiledDocument`, which carries the arena, source-id handle map, and `UiV2ComponentGraph`. The graph records each arena node's stable handle, source id, component name, parent handle, and child handles so editor/runtime consumers can inspect the component graph without depending on a runtime-local compiled type.

`UiV2SurfaceBuilder` consumes the arena directly through `ui::v2::surface_tree` and inserts one flat arena node at a time into the retained runtime tree. The folder-backed `surface_tree` owner splits node insertion, layout parsing, slot projection, interaction inference, and shared parsing helpers so the v2 runtime surface path can evolve without growing the root builder. It does not build a recursive `UiTemplateNode` tree and no longer routes through the legacy `UiTemplateTreeBuilder` or `UiTemplateSurfaceBuilder` path. Node paths are stable flat source paths (`v2/<source_id>`) so very deep UI documents do not allocate path strings proportional to depth squared. The `ui_v2_surface_projection_does_not_call_template_tree_builder` guard scans the v2 surface projection sources and fails if that main path regains old template-builder dependencies.

`surface_tree::interaction` now derives default input capabilities from the formal `UiComponentDescriptorRegistry::editor_showcase()` catalog. Authored `input_*` metadata remains the override path, authored bindings still mark event-specific nodes as interactive, and catalog descriptors fill the default for components that do not repeat those flags in every v2 asset. Runtime input, numeric, selection, reference, and collection components receive click/hover/focus defaults; container and feedback components become interactive only when their descriptor events imply interaction, so `MessageBox`, `Popup`, and `Foldout` can receive input while passive `ProgressBar` stays non-interactive.

`UiV2ComponentInstancer` expands local and imported composite components through an explicit stack. It validates source graph cycles before expansion so a cyclic flat graph fails fast instead of spinning before compiler validation. Component slot fills are grouped by authored `slot.name` / `slot_name`, then checked against `UiV2ComponentDefinition.slots`: required slots must be present, unknown slots are rejected once a component declares slot schema, and non-`multiple` slots reject multiple fills. Components with no declared slot schema continue to accept the default slot as the simple no-schema path.

The instancer treats source children as traversal input, not as output children. Every output node starts with an empty child list and receives children only when the arena instancing pass mounts a child into the output parent. This avoids duplicating child edges when a flat authored view is compiled and prevents the component graph from growing into an invalid or runaway tree. Original v2 node IDs are preserved for non-expanded source nodes so style resolution and diagnostics keep stable source IDs; generated `v2n*` IDs are used only for component-expanded nodes.

UI Asset Editor v2 component authoring uses a legacy-shaped projection only as the editing adapter. Selected-node props/state literal patching is exposed through the session API and then serialized back through the schema-aware v2 save path, so v2 component assets keep `kind = "component"` plus flat `[nodes.*].props/state` instead of downgrading to recursive widget TOML. The same values are flattened as `prop` and `state` rows into both the inspector summary stream and the dedicated retained-host `UiAssetInspectorWidgetData.prop_state_rows` model, where each row carries `kind`, `path`, `value`, and `display`. Retained conversion now has a dedicated `ui_asset_detail_fields.rs` owner that turns Widget, Slot, Layout, and Binding inspector data into dynamic `InputField` nodes. It emits basic widget fields, generic props/state rows, slot mount/padding/layout fields, layout semantic/container fields, and binding id/event/route fields, then grows each inspector section frame before shifting later sections. These generated fields use `dispatch_kind = "commit_only"`: text input mutates the native focus buffer and repaints only the input damage region, while Enter/commit dispatches `ui_asset_detail|<instance>|<detail>|<action>|<index>` through the pane surface dispatcher. Draft edit ids stay under `ui_asset_detail_draft|...`, so keypresses do not trigger UI asset document patches or full presentation rebuilds. The `asset_editor` component adapter accepts the resulting `widget.prop.<path>` / `widget.state.<path>` commit targets, so retained/v2 controls can display and author generic component parameters without parsing generic inspector text or adding one hard-coded editor action per key.

## Style Resolver

`UiV2StyleResolver` compiles v2 style rules into ordered selector rules and applies them by specificity plus declaration order. It supports component selectors, `.class`, `#control_id`, and pseudo states such as `:hover`, `:pressed`, `:focused`, `:disabled`, `:checked`, and `:selected`. Pseudo states are derived from true values in `props` and `state`; `hovered` also aliases to `hover`.

The static resolver intentionally leaves authored inline `style.self` out of the selector cascade. Static non-pseudo rules populate `UiTemplateNodeMetadata.attributes`; authored inline style and final override values populate `UiTemplateNodeMetadata.style_overrides`. Runtime render extraction reads `style_overrides` first and falls back to `attributes`, preserving the highest-priority inline authored value without letting it become a stale pseudo-state baseline. The guard `ui_v2_inline_style_overrides_cascade_values_in_style_overrides` freezes this split with a Button whose selector cascade resolves `button_variant = "contained"` while inline style keeps `button_variant = "outlined"` in `style_overrides`.

Typed style consumers sit on top of the same v2 value maps. `zircon_runtime::ui::style` defines the generic `StyleProperty` / `StyleField<P>` resolver and weak `StyleSheetScope` list used by `ButtonStyleFields`; `resolve_button_style_from_values(...)` is the narrow bridge from runtime/editor TOML value maps to shared `ResolvedButtonStyle`. The first shipped consumer is Material Button/IconButton styling: enum fields such as `button_variant`, `button_color`, `button_size`, `icon_placement`, and `button_interaction_state` are parsed into `zircon_runtime_interface::ui::style` enums instead of being passed through as host-local strings. This layer is deliberately separate from the v2 selector engine so future typed style properties can reuse the same cascade/override mechanics without changing the `.ui.toml` schema.

Token values can be referenced as `$token.name` or `var(token.name)` and are resolved from the document token map before styles are projected into runtime metadata. Resolution walks nested arrays and tables, so Material declarations such as `background = { color = "$material_surface" }` and pseudo-state table declarations are normalized before render extraction reads them.

`UiV2SurfaceBuilder` now splits style projection into a static baseline plus a heap-resident runtime pseudo-state index. Static, non-pseudo rules are baked into `UiTemplateNodeMetadata` when the flat arena becomes a retained `UiSurface`. Pseudo-state rules are kept in `UiV2RuntimeStyleIndex`, with a per-node baseline copy of attributes/style overrides captured before runtime state is applied. Hover, press, focus, checked, disabled, expanded, popup-open, and selected changes in `UiSurfaceComponentStateStore` call back into that index, reuse selector paths from the retained tree with explicit traversal stacks, and overlay only the currently matching pseudo-state declarations. Stateful v2 booleans that have retained surface fields, such as `checked`, `disabled`, and `expanded`, are seeded into `UiTreeNode::state_flags` or component state and then resolved from retained surface state rather than stale TOML props. Descendant pseudo selectors such as `.row:hover Text.label` are resolved while walking the affected subtree, so a parent state change can restyle deep children without rebuilding each child's ancestor path. This lets Material hover/pressed/focus/checked/disabled/expanded variants turn on and off without reloading TOML, reparsing selectors, or permanently baking authored `state.hovered = true` into the node baseline.

Runtime style invalidation stays dirty-domain aware. Render-only visual keys such as `background`, `foreground`, `border`, `outline`, `radius`, `opacity`, elevation, and typed button visual fields (`button_variant`, `button_color`, `button_size`, `button_interaction_state`, and `icon_placement`) mark render dirty only; text-affecting keys mark text dirty, and unknown style keys mark style dirty so correctness wins over a false render-only optimization.

`editor_material.v2.ui.toml` now carries the shared Material pseudo-state rules for common imported v2 assets. `.material-control` and related button, primary button, text button, outlined field, line edit, menu item, list item, table row, and tab classes define hover, pressed, focus, selected, and disabled variants where applicable. Button focus rules are border/ring overlays rather than background overrides, so a pointer down that also focuses the button keeps the stronger pressed fill while still drawing the focus border. Imported views such as Welcome and Component Showcase therefore exercise the runtime style index with real theme rules instead of relying on the local Material demo's authored pseudo-state fixture only. The `editor_material_runtime_pseudo_states_rebuild_render_extract_variants` guard drives the real Welcome startup demo button through pointer move and primary press, then asserts the final render command background and focus border colors. Its rebuild reports stay render-only for these state changes, so the current pseudo-state path is verified from retained component state through style re-resolution into render extraction without layout or arranged-tree rebuilds.

Programmatic property mutation now enters the same runtime pseudo-state path. `UiSurface::mutate_property(...)` keeps the existing tree mutation and focus reconciliation semantics, then syncs accepted `pressed` patches into `UiSurfaceComponentStateStore` and re-applies the runtime style index for pseudo-state-affecting properties such as `checked`, `selected`, `disabled`, `enabled`, and `pressed`. This means Rust callbacks, bindings, and authoring patches can drive `:checked` and `:disabled` Material variants without waiting for a pointer/focus event or reloading v2 TOML. The property mutation guard toggles a v2 `Toggle` through `checked = true` and `enabled = false`, verifies the resolved runtime colors immediately, and confirms the dirty state remains render/input scoped rather than layout scoped.

Pointer release now applies default retained component actions for simple checked controls. `Toggle`, `Checkbox`, `CheckBox`, `Switch`, and `ToggleButton` flip their `checked` state on primary click through `UiSurface::mutate_property(...)`, emit a bound `onChange` `ValueChanged { property = "checked" }` event when present, and immediately feed the runtime pseudo-state resolver. The v2 click guard drives a real `Toggle` through down/up pointer dispatch, checks the emitted change envelope, verifies `:checked` style resolution, and rebuilds only the render extract.

Expanded editor/runtime components use the same retained interaction path. `Group`, `Foldout`, `InspectorSection`, `TreeRow`, and `TreeView` primary clicks flip `expanded` through `UiSurface::mutate_property(...)`, emit a bound `onToggle` `ToggleExpanded` event when present, and update `:expanded` style from the retained component state. The runtime style index now strips retained state keys from its static baseline before reapplying pseudo-state overlays, so authored `expanded = true` can toggle back to false without the baseline restoring the original TOML value and forcing a style/layout rebuild.

Popup-capable selection components now have a typed default click path. `Dropdown`, `ComboBox`, `EnumField`, `FlagsField`, `SearchSelect`, `ContextActionMenu`, and `Popup` flip `popup_open` through `UiSurface::mutate_property(...)`, drive `:open` and `:popup_open` style variants from retained state, and emit only the matching `OpenPopup` or `ClosePopup` binding. This keeps authored open/close routes separate instead of sending a generic `Commit` to every click binding, while pure popup visual changes stay render-only.

Numeric range controls now have a retained default interaction. `RangeField` and `Slider` map primary-click x-position inside the arranged frame to `min..max`, apply authored `step` snapping, patch `value` through `UiSurface::mutate_property(...)`, and emit a bound `ValueChanged { property = "value" }` event. Press starts pointer capture and emits `BeginDrag`, captured move updates continue even outside the hit frame and emit `DragDelta`, and release ends capture with `EndDrag`. Focused range controls consume left/right/up/down navigation as step changes, and accessibility `Increment`/`Decrement` actions on slider roles use the same retained step path. Numeric `value` changes for `RangeField`/`Slider`/`ProgressBar` are classified as render-only metadata mutations so track fill updates do not force layout or hit-grid rebuilds.

The retained default component interaction family now lives in `surface/default_interactions.rs`. `surface.rs` remains the orchestration boundary for pointer, focus, rebuild, render, and accessibility dispatch, while the child module owns Toggle/Checkbox checked toggles, Foldout/TreeView expansion, ComboBox/Popup open state, RangeField/Slider click-drag-step behavior, and the typed component event filtering used by those defaults. This keeps future component interaction productization out of the already large surface coordinator without changing the public `UiSurface` call path.

The surface rebuild pipeline now lives in `surface/rebuild.rs`. That module owns `UiSurfaceRebuildReport`, dirty flag aggregation, dirty-node counting, full rebuild, incremental rebuild, render extract cache refresh, and layout recompute entry points. `surface.rs` keeps the retained surface data structure plus input/focus/pointer orchestration, while rebuild timing and cache accounting have a single owner. The split is mechanical by design: public calls such as `compute_layout`, `rebuild`, `rebuild_dirty`, `dirty_flags`, `clear_dirty_flags`, `mark_node_dirty`, and `apply_pointer_dispatch_dirty` keep the same `UiSurface` API.

The runtime manager render-dirty guard now accounts for both explicit dispatcher dirty requests and retained component-state hover dirtiness. A first pointer move can mark the dispatcher target and newly hovered retained nodes in the same frame; the acceptance condition is therefore the exact de-duplicated dirty source set, while the important rebuild contract remains unchanged: render-only updates rebuild render extraction without layout, arranged-tree, or hit-grid work, and all dirty flags are consumed on the persistent surface.

## Editor Cutover Assets

The first production editor projections now use v2 view and chrome assets through the heap-backed prototype cache. `UiV2PrototypeStoreFileCache` is the v2 file/import/cache layer for this path: it resolves `res://` v2 imports, folds imported style tokens and rules into the root document used for compilation, stores imported component/style documents in the v2 prototype store, and returns an `Arc<UiV2CompiledDocument>` for projection. The cache recognizes `.zui` and `.v2.ui.toml` suffixes case-insensitively so uppercase filesystem names cannot bypass the `.zui` profile validator. When an asset-id import can resolve to both a migrated `.zui` component and a legacy `.v2.ui.toml` document with the same asset id, the index deterministically prefers the `.zui` document before falling back to the legacy file. The current v2 pane set covers ProjectOverview, AssetBrowser, Console, Hierarchy, Inspector, AssetsActivity, AnimationEditor, and Welcome. The current root chrome set covers menu chrome, menu popup, page chrome, dock header, status bar, and activity rail. These files exercise the same v2 DTO/cache path as the demo assets, so bootstrap and boundary tests can guard the hard cut without preserving recursive `UiTemplateNode` as the main path.

Direct v2 view projection now preserves interaction metadata instead of flattening every unknown component into a passive mount. `ViewTemplateNodeData` carries `component_role`, `value_text`, `binding_id`, `edit_action_id`, and `commit_action_id`; `view_projection.rs` derives those fields from authored component ids, `props`, and `events`, while `template_node_conversion.rs` forwards them into the retained host contract. AssetBrowser and AssetsActivity search fields are authored as v2 `TextField` nodes with `AssetSurface/SearchEdited` change bindings, so native hit testing can focus the input and route keyboard edits through `asset_control_changed` without rebuilding the pane or falling back to legacy asset-surface controls. Assets Activity keeps `dispatch_kind = "asset"` for the activity default, while Asset Browser uses `dispatch_kind = "asset:browser"` so text edits retain the browser source through native focus handling. Those search fields keep `placeholder` authored in TOML and let projection derive `value_text` from the live search query: empty query projects an empty input buffer, non-empty query projects the query into both visible text and input state. The UI Asset Editor node projection shares the same resolver path so authored v2 previews keep component roles and binding ids when projected into retained template nodes.

Asset Browser and Assets Activity static toolbar buttons now use the same v2 interaction metadata path instead of a visual-only selected state. Filter chips, view-mode toggles, utility tabs, locate, import, and open-browser buttons carry `dispatch_kind`, canonical `action_id`, static `value`/`value_text`, and `events` in their v2 TOML. Native pointer dispatch recognizes `asset` and `asset:<surface>` dispatch kinds, maps `SetKindFilter`, `SetViewMode`, and `SetUtilityTab` to `asset_control_changed`, and forwards the static value to the existing Rust callback bridge. This keeps browser-specific patches on the `browser` asset surface while preserving the activity surface default for existing controls.

Runtime builtin fixtures are now on the same v2 mainline. `RuntimeUiFixture` maps each logical runtime entry to `zircon_runtime/assets/ui/runtime/fixtures/*.v2.ui.toml`, and `RuntimeUiManager` loads those files through `UiV2PrototypeStoreFileCache -> UiV2SurfaceBuilder -> ui::v2::surface_tree -> UiSurface`. The runtime fixture path has no `include_str!`, `UiAssetLoader`, `UiDocumentCompiler`, `UiTemplateTreeBuilder`, or `UiTemplateSurfaceBuilder` fallback; old schema materializers remain only for migration/test-only legacy surfaces.

Runtime fixture interaction also stays on the retained v2 surface instead of rebuilding the document. `RuntimeUiManager::dispatch_pointer_event(...)` now applies `UiPointerDispatchEffect::RequestDirty(...)` back onto the touched `UiSurface` nodes through `UiSurface::apply_pointer_dispatch_dirty(...)`, then calls `UiSurface::rebuild_dirty(root_size)` only when structured dirty domains are present. Render-only requests rebuild the render extract without layout, arranged-tree, or hit-grid work; layout requests flow through the incremental layout path. Primary press/release also reduces into `UiTreeNode::state_flags.pressed` on the same retained surface, so pressed styling and accessibility state no longer depend on re-projecting v2 TOML. Hover/focus/press flags are mirrored into `UiSurfaceComponentStateStore`; focus sources include autofocus, programmatic focus, pointer focus, keyboard navigation, and focus clearing. A state transition on that store now re-resolves matching pseudo-state rules through the retained runtime style index, overlays the result onto the captured static baseline, and marks only the affected dirty domains. Repeated same-target hover events stay clean after the first repaint. This gives v2 style/render work a heap-resident component state source while leaving authored TOML immutable. This keeps runtime HUD/dialog input aligned with the editor dirty-domain cutover and prevents pointer handlers from re-entering the v2 TOML loader for ordinary component state changes.

The editor-owned runtime preview copies under `zircon_editor/assets/ui/runtime/*.ui.toml` have been removed. Runtime preview coverage in UI Asset Editor now calls `UiAssetEditorSession::from_v2_source(...)`, compiles with `UiV2DocumentCompiler`, and builds the preview through `UiAssetPreviewHost::new_v2(...)`. The session still creates a lightweight legacy-shaped projection document for the existing editor hierarchy/canvas labels, but that projection is not used to build the runtime surface and is guarded so host preview tests keep resolving the runtime crate `.v2.ui.toml` fixtures.

UI Asset Editor v2 authoring preview now uses the same prototype-store compiler boundary for registered imports. `v2_authoring.rs` keeps external component/style documents resident in session import state, folds imported style tokens/rules into the current v2 source, builds a `UiV2PrototypeStore`, and compiles with `compile_with_prototype_store`. Imported composite components therefore fill named slots and receive props/state/class patches in the preview surface without copying prototype internals back into the current source.

Local v2 component assets are now authorable through the same editor session instead of being treated as preview-only prototypes. `lifecycle.rs` projects local `[components.*]` roots into the existing authoring model, `session_state.rs` and `hierarchy_projection.rs` use component roots when a document has no `[root]`, and `v2_authoring.rs` synthesizes a transient preview root from the first component root only for preview compilation. Canonical save preserves the v2 `component` asset shape, including `[components.*]`, slot declarations, and flat `[nodes.*]`. The authoring projection validates missing component roots, missing child nodes, and component child cycles before building editor state, so malformed component assets fail with explicit v2 asset errors instead of silently disappearing from the editor tree. Editor asset reference analysis also handles first-class v2 view/component/style payloads through their runtime direct-reference contracts.

`component_showcase.v2.ui.toml` is the first runtime component-catalog window hard-cut to the same path. It is a flat v2 view asset whose remaining inline nodes are shell/layout nodes instead of inline demo-control sections. It now imports `.zui` component prototypes for the Unreal-style command toolbar, left category navigation, Visual/Feedback gallery section, Input/Numeric gallery section, Selection/References gallery section, Collections/Inspector gallery section, right retained-state panel, and bottom log (`showcase_command_toolbar.zui#ShowcaseCommandToolbar`, `showcase_category_nav.zui#ShowcaseCategoryNav`, `showcase_visual_section.zui#ShowcaseVisualSection`, `showcase_input_section.zui#ShowcaseInputSection`, `showcase_selection_section.zui#ShowcaseSelectionSection`, `showcase_collections_section.zui#ShowcaseCollectionsSection`, `showcase_state_panel.zui#ShowcaseStatePanel`, and `showcase_bottom_log.zui#ShowcaseBottomLog`) while keeping the full view document in `.v2.ui.toml`. The category nav prototype owns the `ShowAllCategory`, Visual, Feedback, Input, Numeric, Selection, Reference, and Collections route bindings so category filtering remains event-driven after the extraction. The Visual/Feedback prototype owns the display-only catalog controls (`Label`, `RichLabel`, `Image`, `Icon`, `SvgIcon`, `Separator`, `ProgressBar`, `Spinner`, `Badge`, and `HelpRow`) while preserving the same demo control ids for projection and filtering. The Input/Numeric prototype owns the button, toggle, text, number, range, color, and vector controls while preserving `UiComponentShowcase/*` event routes for commit, value change, toggle, and drag bindings. The Selection/References prototype owns dropdown, combo, enum, flags, search-select, asset, instance, and object reference controls while preserving option and reference event routes. The Collections/Inspector prototype owns group, foldout, property row, inspector section, array, map, list/table/virtual/paged rows, world-space surface, tree row, and context action menu controls while preserving collection, tree, popup, and paging routes. The state-panel prototype owns the selected-category, last-control/action, current-value, validation, drag/drop, and event-log rows while keeping retained overlay data driven by the existing host runtime state. Old showcase and Material meta-component inputs remain only under `zircon_editor/src/tests/fixtures/ui_legacy/**` as legacy authoring/migration fixtures; the builtin document registry now points at v2 and guard tests reject `component_ref` / old Material meta imports on the showcase main path.

The editor showcase catalog also declares common runtime projection metadata props (`layout_*`, `input_*`, text metrics, and Material variant fields) so direct v2 catalog nodes retain the measurement and interaction semantics previously supplied by recursive Material meta components.

Retained-host v2 assets must author the same runtime projection metadata on the hot path instead of relying on editor-only expansion. `inspector_surface_controls.v2.ui.toml` now projects an `InspectorSurfaceControls` root with direct retained control children, keeps field labels as metadata, authors explicit `placeholder` / `value` / `value_text` on input fields, authors `layout_min_height` and `input_*` flags on interactive roots, and uses direct `NumberField` / `Button` nodes for the inspector transform and action controls. `RetainedUiHostAdapter` projects visible text from non-empty `text`, then falls back to `label` only when the node does not also carry field/value metadata such as `placeholder`, `value`, `value_text`, `options`, `items`, or `entries`; this preserves action labels while preventing input labels from becoming typed text.

## Taffy Bridge

`zircon_runtime::ui::layout::taffy_bridge` imports Taffy into runtime and converts the Taffy-owned subset of Zircon layout contracts into `taffy::style::Style`. Flex, Grid, Wrap, and Size-style layout roles are routed toward Taffy. Overlay, Canvas, Popup, VirtualList, and editor docking remain Zircon-owned.

`layout/pass/taffy_arrange.rs` is called from the global arrange pass for safe Flex, Wrap, and Grid subtrees. The current integration stage allows v2-authored nodes with template metadata to use Taffy because metadata is render/event data, not a layout placement contract. Authored slot frame policies, canvas/grid placement overrides, or non-default anchor/pivot/position still fall back to Zircon arrangement so absolute placement, Overlay, Canvas, Popup, VirtualList, and editor docking semantics remain Zircon-owned. The guard tests prove Overlay and Scrollable/VirtualList semantics stay outside the Taffy path, while `taffy_layout_pass_accepts_template_metadata_from_v2_assets` covers the v2 template cutover.
