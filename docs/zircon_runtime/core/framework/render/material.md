# Render Material Contracts

## Purpose

`zircon_runtime::core::framework::render::material` owns neutral material DTOs. Asset-side `MaterialAsset` converts into these descriptors, while concrete WGPU bind groups, pipeline keys, runtime fallback material preparation, and renderer stats remain under `zircon_runtime::graphics`.

The top-level `zircon_runtime::core::framework::render` facade re-exports the material DTOs that resource streaming and renderer diagnostics consume. `RenderMaterialPropertyUniformSummary` must remain available through that facade because `resource_streamer_accessors.rs` depends on the neutral render-framework surface instead of reaching into the child material module directly.

## Product Surface

`StandardMaterialDescriptor` carries the M3A PBR-ready material surface: shader and texture dependencies, base color, normal, metallic/roughness, metallic-roughness texture, occlusion, emissive data, alpha behavior, unlit flag, double-sided flag, and fallback policy.

`ColorMaterialDescriptor` provides the simple unlit color/texture material contract for 2D and fallback paths. It shares the same dependency, alpha, double-sided, and fallback fields so later Core2d/Core3d classification can consume a common material shape.

`RenderMaterialDependencySet` stores the required shader and deduplicated texture references. `MaterialAsset::direct_references()` projects those dependencies into the normal imported-asset dependency path.

`RenderMaterialReadinessReport` is the structured acceptance result for asset readiness. It carries the material name, dependency set, fallback policy, validation errors, fallback usage records, optional prepared property, uniform, standard texture-slot, and non-standard texture-slot summaries, and non-blocking diagnostic rows instead of relying on silent fallback-only behavior. `RenderMaterialReadinessSummary` is a derived compact view over that report; it exposes the ready/fallback/diagnostic booleans, validation/fallback/diagnostic counts, and the four prepared material summaries without becoming a second source of readiness truth. `RenderMaterialIssueState` is the sibling issue aggregate: it copies validation errors, fallback usage records, and non-blocking diagnostic rows for panels that want the problem list without prepared material rows. `RenderMaterialPreparedState` is the matching detail aggregate: it copies the report's property-value rows, uniform field/unsupported rows, and standard/non-standard texture slot rows into one DTO for panels that need the prepared material surface but not validation/fallback issue lists. The `management` module owns the list-oriented management DTOs derived from those readiness views, with focused child modules for record summaries, status/issue indexes, views, query execution, query filters, query controls, selection, page helpers, result state, and result actions. `RenderMaterialManagementSnapshot` combines summary, issue-state, and prepared-state for callers that want one panel payload while keeping the full readiness report as the owner of all source rows. `RenderMaterialManagementRecord` wraps that snapshot with the material `ResourceId` and optional material name so list/table consumers do not need to rejoin identity after fetching the management payload. `RenderMaterialManagementRecordSummary` counts records by total, ready, diagnostic, fallback, and invalid state for panel headers and filters, and it also carries the list-level `RenderMaterialReadinessStatus` derived from those counts; its declaration, issue-row total derivation, and shared issue-kind helper live in `management/record_summary.rs` while the root management module keeps the public re-export. `RenderMaterialManagementStatusIndex` buckets record ids by readiness status for filter tabs and badge navigation. `RenderMaterialManagementStatusView` pairs one selected readiness status with the matching material ids and compact overview rows so filtered panels can fetch the selected bucket without scanning every overview row. `RenderMaterialManagementOverviewRecord` and `RenderMaterialManagementOverview` are the compact list form: they keep only material id/name, readiness summary, collection summary, and the same status index for first-pass tables that do not need issue or prepared detail rows. `RenderMaterialManagementSelection` is the full-record selection view for detail panes; it preserves requested material-id order after duplicate ids are collapsed, reports selected records with derived summary/status buckets, and keeps missing ids explicit for stale UI selections. Its declaration and duplicate-collapsing helpers live in `management/selection.rs`, while query-page/detail bridging remains in `management/query_selection.rs`.

`RenderMaterialManagementIssueIndex` is the issue-type companion to the status index. It buckets ordered material ids by validation-error rows, fallback-usage rows, and diagnostic rows, and it is attached to record sets, compact overviews, query results, and selections. A material has exactly one readiness-status bucket, but it may appear in more than one issue bucket when a report has multiple issue-row classes; the index is therefore for drilldown navigation, not for deriving unique material counts.

`RenderMaterialManagementSortKey`, `RenderMaterialManagementSortDirection`, and `RenderMaterialManagementSortOrder` define the neutral management-list sort contract for material id, material name, and readiness status. Sorted record sets, overviews, and status views rebuild the status index or filtered id list from sorted rows, so filter buckets follow the same display order a panel shows.

`RenderMaterialManagementQuery`, `RenderMaterialManagementQueryState`, `RenderMaterialManagementQueryFilter`, `RenderMaterialManagementQueryResultState`, `RenderMaterialManagementQueryResultActions`, `RenderMaterialManagementQueryControls`, `RenderMaterialManagementPageRequest`, `RenderMaterialManagementPageInfo`, `RenderMaterialManagementPageWindow`, and `RenderMaterialManagementQueryResult` add the shared query contract for material management tables. `management/query.rs` owns the query/result declarations plus status, issue-kind, text, sort, and page execution over compact overview rows. `management/page.rs` owns the page request/info declarations and their basic construction, while `management/page_navigation.rs` owns adjacent-page and display-window derivation. The query can filter by readiness status and issue kind, match text case-insensitively against material name or `ResourceId`, reuse the same sort order, and return a page of compact overview rows. The query state normalizes the optional text filter and exposes active-filter/page flags for panel filter badges, clear actions, and first-page resets without scanning rows. The active-filter rows keep status, issue-kind, and text filter chips in display order and attach the exact query needed to remove each filter while preserving sort and page size and returning to the first page. The query-result state classifies table chrome states such as no records, filters returned no records, an intentionally empty page, a page offset outside the filtered set, or a populated page. The query-result actions read the same state and page metadata to derive optional clear-filter, first-page, previous-page, and next-page queries for panel controls. The query-controls aggregate packages query state, active filters, result state, actions, page metadata, page window, and display page numbers into one DTO for panels that want one chrome read. Query-result summary, status index, and issue index describe the filtered result set before pagination, while `records`, `page.returned_count`, and the derived page window describe only the current page.

`RenderMaterialManagementQueryFacet` and `RenderMaterialManagementQueryFacets` are the candidate-filter companion for the same query result. They derive ordered readiness-status and issue-kind facet rows from the filtered pre-pagination status and issue indexes, carry counts plus active flags, and include the exact query needed to select each facet while preserving the other filters, sort order, and page size. `RenderMaterialManagementQueryControls` carries the facets beside active-filter rows and paging chrome so panels can render filter menus without rescanning page-local rows.

`RenderMaterialManagementSelection` and `ResourceStreamer::material_management_selection(...)` complement query pages with full management rows for selected ids. Selection is derived from the current record set, preserves request order, collapses duplicate ids, and returns `missing_material_ids` rather than hiding selections that disappeared after a refresh.

`RenderMaterialDiagnosticSource` tags validation and diagnostic records with the layer that produced them: material asset metadata, shader schema, shader payload readiness, WGSL capture, material override, texture slot, renderer material ABI, material uniform preparation, or dependency resolution. The enum is intentionally neutral render-framework data so asset import, runtime streaming, and editor projection can show the same typed diagnostics without passing runtime/editor objects across the boundary.

`RenderMaterialPropertyValue` is the neutral typed value payload for shader schema properties. It covers bool, float, signed and unsigned integers, string, vec2, vec3, and vec4/color values. `RenderMaterialPropertyValueSummary` is the companion inspection DTO for the projected map; it records total values, per-kind counts, uniform-eligible values, and non-uniform values such as strings before any byte encoding occurs. `RenderMaterialPropertyValueState` is the detail row for the same projection: it carries the property name plus typed value and can report whether that row can be encoded into the fixed uniform payload. The value types and summaries are not a GPU uniform layout and do not imply a bind-group ABI; they are the renderer-facing value projection that lets later uniform-buffer or reflection work consume material properties without reparsing TOML.

`RenderMaterialPropertyUniformPayload` is the CPU-side preparation step for shader property bytes. It converts the deterministic shader property value map into a byte buffer plus field layout rows using scalar/vector alignments that the graphics resource streamer can upload. String properties are retained as unsupported entries instead of being silently dropped, and `unsupported_diagnostics()` projects those rows into non-blocking `MaterialUniform` readiness diagnostics. After streaming, the full readiness report mirrors the encoded field rows in `uniform_fields` and the unsupported rows in `uniform_unsupported`, while the compact readiness summary remains count-only. They remain authoring/runtime metadata until a separate binding path exists for non-uniform property kinds. This core DTO still does not own WGPU objects or infer a complete shader reflection ABI.

`RenderMaterialTextureSlotSummary` is the matching compact inspection DTO for authored material texture slots after resource resolution. The same DTO is used for fixed PBR slots (`base_color`, `normal`, `metallic_roughness`, `occlusion`, and `emissive`) and shader-specific non-standard slots. It counts authored slots, how many resolved to concrete texture ids, and how many remain fallback-only. `RenderMaterialTextureSlotState` is the companion detail row: it carries the authored slot key plus the resolved `ResourceId` when upload/lookup succeeded, or `None` when the authored slot fell back. Summary stays count-only; state rows are for editor/runtime detail panels that need slot-key-level inspection without reading `MaterialRuntime`.

M5A wires the `StandardMaterialDescriptor` surface into the runtime resource streamer. `MaterialRuntime` now stores base color, emissive color, metallic, roughness, double-sided, alpha, unlit, standard PBR texture ids, shader property values projected from shader schema, non-standard shader texture slot ids, a `PipelineKey`, and the readiness report that was produced while resolving material, shader, and texture dependencies.

`PipelineKey` is intentionally renderer-owned. It includes shader identity/revision, double-sided state, alpha-blend/alpha-mask state, alpha cutoff bits, unlit state, and authored StandardMaterial texture-slot presence bits. The texture presence bits come from authored descriptor references rather than only successfully uploaded GPU texture ids, so a KTX/container texture that falls back still compiles the same material variant requested by the asset.

## Validation

`RenderMaterialValidationError::InvalidMaskCutoff` records invalid `AlphaMode::Mask` values. The accepted range is finite `0.0..=1.0`.

`RenderMaterialReadinessDiagnostic` carries material-local diagnostic strings that should remain visible without marking the material unavailable. `MaterialAsset.validation_diagnostics` entries are projected into this non-blocking list with `RenderMaterialDiagnosticSource::MaterialAsset` and stable paths such as `material.validation_diagnostics[0]`. `RenderMaterialReadinessReport::is_ready()` still depends only on validation errors and fallback usage.

`RenderMaterialReadinessStatus` is the derived display classification for the same issue rows. Its severity order is `Invalid` when validation errors exist, `Fallback` when fallback usage exists without validation errors, `Diagnostic` when only non-blocking diagnostics exist, and `Ready` when no issue rows exist. `RenderMaterialReadinessReport::status()`, `RenderMaterialIssueState::status()`, and `RenderMaterialReadinessSummary.status` all use this same order, so panels do not need to infer a badge state from three counters.

`RenderMaterialValidationError::UnresolvedShaderReference` and `UnresolvedTextureReference` record dependency resolution failures when the caller supplies resolver functions through `MaterialAsset::readiness_report_with_resolution(...)` or when the resource streamer cannot load the declared locator as the expected typed asset. The matching `RenderMaterialFallbackUsage` entries record whether the fallback was for a shader or a named texture slot. `RenderMaterialReadinessReport::is_ready()` now requires both no validation errors and no fallback usage.

`RenderMaterialValidationError::UnknownPropertyOverride`, `PropertyOverrideTypeMismatch`, and `UnknownTextureSlot` are emitted by `MaterialAsset::shader_contract_diagnostics(...)` when a `.zmaterial` instance does not match the loaded `.zshader` contract. Unknown overrides are sourced from `MaterialOverride`, wrong override types from `ShaderSchema`, and unknown texture slots from `TextureSlot`; each error preserves a stable document path such as `overrides.base_color` or `textures.albedo` for editor highlighting.

`ResourceStreamer::ensure_material(...)` stores those shader/material contract errors on the runtime readiness report after loading the referenced shader contract. `ResourceStreamer::material_readiness_status(...)` exposes the derived `RenderMaterialReadinessStatus` directly for badge-only callers, `ResourceStreamer::material_readiness_summary(...)` derives the compact `RenderMaterialReadinessSummary` from the stored report for panels and stats paths that only need status, counts, and prepared summary DTOs, `ResourceStreamer::material_issue_state(...)` derives `RenderMaterialIssueState` for callers that want validation/fallback/diagnostic rows as one issue payload, `ResourceStreamer::material_prepared_state(...)` derives `RenderMaterialPreparedState` from the same report for callers that want all prepared detail rows in one read, and `ResourceStreamer::material_management_snapshot(...)` derives `RenderMaterialManagementSnapshot` when a panel wants all three derived views together. `ResourceStreamer::material_management_record(...)` adds the requested material id and report-carried material name around that snapshot for single-row consumers, while `ResourceStreamer::material_management_records(...)` returns all prepared material records sorted by `ResourceId` for stable panel tables. `ResourceStreamer::material_management_record_set(...)` wraps that same stable list with its derived collection summary and status index so table callers can fetch rows, header badges, and filter buckets together. `ResourceStreamer::material_management_overview(...)` derives a lighter list payload from the same record-set path for first-viewport tables that only need id/name/status/summary rows. `ResourceStreamer::material_management_status_index(...)` returns only the derived status buckets for callers that already have row data, while `ResourceStreamer::material_management_status_view(...)` returns the matching compact overview rows for one selected status. `ResourceStreamer::material_management_issue_index(...)` returns only the derived issue buckets, while `ResourceStreamer::material_management_issue_view(...)` returns the matching compact overview rows for one validation, fallback, or diagnostic issue kind. `ResourceStreamer::material_management_query_selection(...)` composes the query result and full-record selection for the current page in one read, keeping the compact page order and the selected detail records aligned. `ResourceStreamer::material_management_record_summary(...)` now reads the summary from the same record-set path for callers that only need collection counts. `render_product_streamer_reports_shader_material_contract_diagnostics` locks the combined runtime surface for unknown property overrides, override type mismatches, missing required shader properties, unknown texture slots, and required texture slots that only provide fallback metadata without a concrete texture reference.

`ResourceStreamer::material_management_record_set_sorted(...)`, `material_management_overview_sorted(...)`, `material_management_status_view_sorted(...)`, and `material_management_issue_view_sorted(...)` accept `RenderMaterialManagementSortOrder` so editor/runtime callers can request id, name, or status ordering from the derived management boundary. Callers that do not pass a sort order keep the existing ResourceId-stable list.

`RenderMaterialValidationError::MissingWgslCapture` records declared shader properties or texture slots that were not found by the lightweight WGSL source scan. Compound `.zshader` import stores those misses on `ShaderAsset.validation_diagnostics`, and `MaterialAsset::readiness_report_with_shader_contract(...)` folds them into the material readiness report with `WgslCapture` as the diagnostic source.

`RenderMaterialValidationError::ShaderReadinessDiagnostic` records shader payload readiness rows that do not have a narrower material error shape. Material readiness currently uses it for invalid entry-point stage tokens and duplicate or empty normalized shader definitions from `ShaderAsset::readiness_report()`. The row keeps the original diagnostic string plus a stable path such as `entry_points.fs_main` or `shader_defs.USE_UNLIT`.

`RenderMaterialDiagnosticSource::MaterialUniform` records renderer uniform preparation notes that should not block material readiness. The current producer is `RenderMaterialPropertyUniformPayload::unsupported_diagnostics()`, which reports unsupported non-uniform property kinds such as strings with stable paths like `uniform.debug_label` after the typed property map has been projected. The matching unsupported detail row remains available through `RenderMaterialReadinessReport.uniform_unsupported` for panels that need the original name/reason pair instead of only a diagnostic string.

The resource streamer also folds authored shader pipeline-layout compatibility into `ShaderReadinessDiagnostic` when a `ShaderAsset` declares bind groups, including bind groups persisted from compound `.zshader` `[pipeline_layout]` TOML. The current mesh renderer material ABI is fixed at group 3 binding 0 as a uniform buffer; a missing group 3, missing binding 0, wrong binding resource type, compute-only visibility, duplicate descriptors, or extra group-3 material bindings is reported with stable paths such as `pipeline_layout.group3` and `pipeline_layout.group3.binding0`. Empty shader pipeline layouts remain accepted because the current renderer can still use its fixed fallback layouts without serialized reflection. The asset-owned `ShaderReadinessReport` separately exposes a layout summary for inspection; the renderer-owned ABI check remains the only path that decides whether authored material bindings conflict with the current mesh renderer contract.

`RenderMaterialValidationError::MissingRuntimeShaderSource` is emitted by the streamer when a declared shader asset exists but cannot provide runtime WGSL. The streamer stores the report on `MaterialRuntime` before returning the blocking material error so renderer/debug consumers can inspect the cause.

`RenderMaterialValidationError::TextureNotUploadReady` records texture payloads that are valid assets but cannot be uploaded by the current texture upload plan. RGBA8 payloads must declare an uploadable decoded format such as `rgba8unorm` or `rgba8unorm_srgb`; descriptor formats that require conversion, such as `rgba16float`, remain valid asset metadata but fall back at material preparation until a conversion path exists. Container payloads keep their `RenderImageDescriptor` metadata and can upload when their DDS/KTX/KTX2/ASTC format, payload layout, single-level/single-layer upload shape, and device support produce a compressed upload plan. Unsupported container shapes such as mip chains are reported before GPU feature support so material fallback reasons stay stable across devices, and unsupported containers still use the default texture fallback with the exact upload-readiness reason preserved in the material report.

`RenderMaterialValidationError::UnresolvedMaterialReference` records a missing material handle at runtime. The resource streamer resolves that case to `builtin://missing-material`, inserts `RenderMaterialFallbackReason::Material`, and keeps the fallback runtime under the originally requested material id so mesh submission can continue while stats and diagnostics remain tied to the missing handle.

`RenderMaterialReadinessReport::push_validation_error_once(...)`, `push_fallback_usage_once(...)`, and `push_diagnostic_once(...)` keep merged streamer diagnostics stable when one material resolution path reports the same shader, texture, material fallback, or local material diagnostic more than once. `MaterialUniform` rows use the same DTO-level de-duplication, so repeated unsupported uniform entries such as `uniform.debug_label` do not inflate readiness diagnostics or submit stats.

`MaterialAsset::readiness_report_with_shader_contract(...)` composes the normal dependency-resolution report with shader-contract diagnostics and shader payload readiness. Missing runtime WGSL is folded into the existing blocking `MissingRuntimeShaderSource` row; imported WGSL capture diagnostics stay `MissingWgslCapture`; invalid entry-point stages and shader definition problems are folded into `ShaderReadinessDiagnostic`. This makes schema mismatches and shader payload gaps readiness failures without turning `.zmaterial` or `.zshader` import into a hard failure, which keeps authoring tools able to open broken material assets and show actionable rows.

## Runtime PBR Baseline

M5A makes StandardMaterial fields visible to the concrete mesh path without moving asset or editor ownership into graphics. `resource_streamer_ensure_material.rs` reads the descriptor, resolves every PBR texture slot, resolves non-standard material texture slots declared outside the fixed PBR aliases, projects shader schema properties into typed runtime values, resolves shader runtime source, records fallback diagnostics, and builds a `MaterialRuntime` plus `PipelineKey`. Missing textures and missing shaders can degrade to default fallback resources; invalid alpha-mask cutoff and missing runtime WGSL remain blocking validation errors.

Shader-declared standard texture aliases are folded into that fixed PBR runtime surface before non-standard slot preparation. If a loaded shader declares `albedo` or `diffuse`, `MaterialAsset::standard_material_descriptor_for_shader(...)` can promote the matching `[textures.<slot>]` reference into the base-color texture field, rebuild the descriptor dependency set from the promoted standard texture fields, and `ResourceStreamer::ensure_material(...)` treats the alias as standard instead of also storing it in `MaterialRuntime.non_standard_texture_slots`. `render_product_streamer_bridges_shader_standard_texture_alias_into_pbr_slot` locks the runtime bridge by proving an `albedo` slot overrides a pre-existing legacy base-color texture and keeps the material ready without duplicate non-standard slot state. `render_product_streamer_shader_standard_alias_shadows_unresolved_legacy_texture` covers the dependency/readiness edge case where the promoted shader alias is valid and a stale legacy `base_color_texture` no longer contributes an unresolved-texture fallback.

Shader property values are runtime prepared state, not uniform-buffer encoding yet. `MaterialAsset::shader_property_values_for_shader(...)` walks the loaded shader property schema, takes a material override first and then the shader default, converts supported TOML bool/number/string/vector rows into `RenderMaterialPropertyValue`, and stores the result on `MaterialRuntime.shader_property_values`. `RenderMaterialPropertyValueSummary::from_values(...)` gives runtime/editor code a compact count of the projected map before uniform payload encoding, `ResourceStreamer::material_property_value_summary(...)` exposes that count without requiring callers to inspect `MaterialRuntime` internals, and `RenderMaterialReadinessReport.property_value_summary` carries the same optional summary beside the uniform summary once resource preparation has run. `RenderMaterialReadinessReport.property_value_states` and `ResourceStreamer::material_property_value_states(...)` expose the matching property-name detail rows, so panels can show concrete bool/float/vector/string values and uniform eligibility without reading `MaterialRuntime`. A panel can therefore distinguish "the renderer received four shader values, three can become uniform bytes, and one remains non-uniform metadata" from later upload state, then drill into the exact property rows. Unknown or mistyped overrides remain readiness diagnostics from the shader-contract validation path; the projection only forwards values that match supported schema kinds.

Material property uniform payloads are prepared from the same map during `ResourceStreamer::ensure_material(...)` and stored on `MaterialRuntime.shader_property_uniform_payload`. Numeric and vector rows produce stable byte ranges and field metadata; unsupported rows such as strings are reported inside the payload and merged into the material readiness report as non-blocking `MaterialUniform` diagnostics so later reflection work can make an explicit decision. Both material overrides and shader schema defaults enter this path after `MaterialAsset::shader_property_values_for_shader(...)` projects them into typed values. `ResourceStreamer::material_readiness_report(...)` exposes those rows with stable `uniform.<name>` paths and now stores `RenderMaterialPropertyUniformSummary`, `uniform_fields`, and `uniform_unsupported` directly on the report after resource preparation, so editor/runtime panels can read payload byte length, encoded field count, unsupported row count, field names/kinds/offsets, and unsupported reasons from the same readiness DTO. `ResourceStreamer::material_uniform_summary(...)`, `material_uniform_fields(...)`, and `material_uniform_unsupported(...)` remain direct accessors for callers that want the compact uniform record or detail rows without inspecting `MaterialRuntime`; the existing individual accessors still expose backing buffer length and legacy count queries. The streamer now also creates a renderer-owned `GpuMaterialUniformResource` per prepared material, backed by a WGPU uniform buffer and bind group with a 64-byte minimum allocation for the current neutral material binding contract. Submit-level stats preserve those unsupported rows through `RenderStats.last_material_diagnostic_count`, so a material can remain ready while still reporting that a string property was retained but not uploaded.

When the referenced shader carries an authored `pipeline_layout`, `resource_streamer_validate_material_shader_layout.rs` checks that the material section matches the renderer's fixed group-3 uniform contract before the runtime material report is stored. Because compound `.zshader` import now stores `[pipeline_layout]` on `ShaderAsset.pipeline_layout`, these diagnostics operate on real authored shader packages as well as programmatic shader fixtures; `render_product_streamer_reports_imported_zshader_material_layout_abi_diagnostics` covers the imported package path. `ShaderAsset::readiness_report()` now exposes the same authored layout shape as context, but does not run this renderer ABI policy. These diagnostics are non-blocking readiness rows for now; they make an incompatible custom `.zshader` visible before later pipeline creation or shader reflection work tries to consume unsupported material bindings.

Standard PBR material texture slots also have runtime readiness counters and detail rows. `ResourceStreamer::material_standard_texture_slot_summary(...)` exposes total/resolved/fallback counts for authored fixed PBR texture references, and `RenderMaterialReadinessReport.standard_texture_slot_summary` carries the same DTO once material preparation has run. `ResourceStreamer::material_standard_texture_slot_states(...)` and `RenderMaterialReadinessReport.standard_texture_slot_states` expose the matching authored slot keys and resolved ids. A standard slot counts as authored only when the descriptor carried a concrete texture reference; unsupported upload payloads, descriptor conversion requirements, compressed mip-chain shapes, and unresolved references remain visible as fallback entries without changing the fixed PBR runtime surface. The fallback detail keeps the original texture reference plus the same upload-readiness message as `TextureNotUploadReady`, so panels do not have to reverse-map the validation error to explain why a slot fell back.

Non-standard material texture slots are runtime prepared state, not full shader reflection yet. `MaterialRuntime.non_standard_texture_slots` records the resolved `ResourceId` for ready slot textures and keeps `None` for slots that fell back, so later renderer bind-group and shader-reflection work can distinguish "slot existed but fell back" from "slot was never authored" without reparsing `.zmaterial`. `ResourceStreamer::material_texture_slot_summary(...)` exposes the compact total/resolved/fallback counts for shader-specific slots, while `ResourceStreamer::material_texture_slot_states(...)` exposes the slot-key detail rows. `RenderMaterialReadinessReport.texture_slot_summary` and `RenderMaterialReadinessReport.non_standard_texture_slot_states` carry the same data once material preparation has run. The split `texture_slot_diagnostics` regression module now covers unsupported uploaded textures, RGBA8 descriptor conversion fallback, compressed mip-chain fallback, and unresolved custom slot references; the custom-slot paths preserve the shader slot key in `RenderMaterialValidationError`, `RenderMaterialFallbackUsage`, `MaterialRuntime.non_standard_texture_slots`, and readiness report slot states.

`RenderMaterialPreparedState` aggregates the prepared material management surface after those four lanes have populated the readiness report. It carries the property projection summary plus named property rows, the uniform payload summary plus field/unsupported rows, and both standard and non-standard texture slot summaries plus slot-key rows. The aggregate is intentionally derived from `RenderMaterialReadinessReport::prepared_state()` rather than stored separately on `MaterialRuntime`, so readiness detail rows remain the source of truth and editor/runtime panels can choose between the full report, the compact readiness summary, or a prepared-state-only snapshot.

`RenderMaterialIssueState` is the matching issue-only snapshot. It is derived from `RenderMaterialReadinessReport::issue_state()` and carries only validation errors, fallback usages, and diagnostics; its helper methods mirror the report's ready/fallback/diagnostic/status semantics. This lets editor/runtime panels fetch the issue list and the prepared-state aggregate independently without duplicating readiness ownership on `MaterialRuntime`.

`RenderMaterialManagementSnapshot` is the combined management view. It is derived from `RenderMaterialReadinessReport::management_snapshot()` and carries `summary`, `issue_state`, and `prepared_state` together; the summary includes the same readiness status as the report and issue-state helpers. `RenderMaterialManagementRecord` is the resource-keyed row form of that snapshot: `RenderMaterialReadinessReport::management_record(...)` takes the externally owned material id, copies the report's optional material name, and embeds the same snapshot. `RenderMaterialManagementRecordSummary` is the collection-level companion derived from a record slice; it exposes total, ready, diagnostic, fallback, invalid, degraded counts, collection-level validation-error/fallback-usage/diagnostic issue-row totals, and the worst list-level status without changing individual record semantics. `management/record_summary.rs` owns that declaration, the issue-row accumulation helpers, and the shared readiness-summary issue-kind predicate used by query and issue-view filtering. The status uses the same priority as material readiness: invalid rows win, then fallback rows, then diagnostic rows, otherwise the list is ready. Issue-row totals are summed from each row's `RenderMaterialReadinessSummary`, so a panel can show how many validation, fallback, or non-blocking diagnostic rows exist without expanding every record. `RenderMaterialManagementStatusIndex` is the matching filter index: it carries the ordered material ids in ready, diagnostic, fallback, and invalid buckets, preserving the same ordering as the source record list. `RenderMaterialManagementStatusView` is the row-bearing companion to that index: it carries the chosen status, the ordered matching ids, and compact overview rows derived from the same records or overview payload. `RenderMaterialManagementIssueView` is the row-bearing companion to the issue index: it carries one selected issue kind, the ordered matching ids, and compact overview rows for materials with validation-error, fallback-usage, or diagnostic issue rows. `RenderMaterialManagementRecordSet` carries the ordered record list, summary, status index, and issue index in one payload. `RenderMaterialManagementOverview` is derived from the same records or record set and carries compact rows with only identity and readiness summary plus the same status and issue indexes; it is meant for table headers and first-pass list rows, not for prepared-state inspection. `RenderMaterialManagementSelection` lets a detail pane request full records for selected material ids after reading a compact page; it reports duplicate-collapsed request count, selected records in request order, missing ids, and selected-record summary/status/issue buckets. `RenderMaterialManagementQuerySelection` is the list-page/detail bridge: it keeps the original query, the compact `RenderMaterialManagementQueryResult`, and a full-record `RenderMaterialManagementSelection` for exactly the ids visible on that page in display order. `RenderMaterialManagementQueryState` is the matching controls read model: it keeps normalized status, issue-kind, text, sort, and page inputs plus booleans for status/issue/text filters, any active filter, and explicit paging. `RenderMaterialManagementQueryFilter` is the row/chip companion to that state: `active_filters()` returns one row for each active status, issue-kind, or text filter and gives each row its own remove-query payload. `RenderMaterialManagementQueryResultState` is the matching result read model: it combines that query state with the query result page counts and classifies empty-record-set, empty-filtered-set, empty-page, page-out-of-range, and populated-page cases for table empty states and reset controls. `RenderMaterialManagementQueryResultActions` is the action companion: it keeps that result state and optional query payloads for clear filters, first page, previous page, and next page so panels do not rebuild query mutations from page-local rows. `RenderMaterialManagementQueryControls` combines that control surface with active-filter rows, the returned page info, page window, display start/end indexes, and page number totals for panels that want one table-chrome DTO. These derived views reduce panel-side fan-out over multiple resource-streamer calls; they do not store additional state on `MaterialRuntime`, and they should not replace the full report for workflows that need dependency identity, fallback policy, or future raw readiness fields.

The record-list module boundary is split by payload role. `management/record_set.rs` owns `RenderMaterialManagementRecordSet`, the ordered full-record list payload, and its overview/status/issue/query/selection entry helpers. `management/overview.rs` owns `RenderMaterialManagementOverviewRecord`, `RenderMaterialManagementOverview`, compact-row status helpers, and compact overview derivation/sorting/querying for table headers and first-pass list rows that do not need prepared-state detail.

The same list payloads also carry `RenderMaterialManagementIssueIndex` and derive `RenderMaterialManagementIssueView` from records or compact overview rows. Unlike `RenderMaterialManagementStatusIndex`, issue buckets are not mutually exclusive: a single invalid material can still appear under diagnostics if the report also has non-blocking diagnostic rows. Query and selection issue indexes are scoped to the filtered or selected set before pagination and after duplicate selection ids are collapsed, matching their summary issue-row totals. Issue views preserve the source display order, or the requested sorted order, so a panel can render one issue bucket without scanning every overview row.

Management sorting is also a derived view. The sort key/direction/order contract and comparison helpers live in `management/sort_order.rs`, while the root management module keeps the public re-export for callers. Name sorting keeps unnamed rows after named rows in both directions, status sorting uses the display priority `Ready`, `Diagnostic`, `Fallback`, `Invalid`, and identity remains the deterministic tie-breaker. Sorting never mutates `MaterialRuntime`; it returns a freshly derived record set, overview, status view, or issue view with summary/index data rebuilt from the sorted row list.

Management query results build on the same derived row model. `RenderMaterialManagementRecordSet::query(...)`, `RenderMaterialManagementOverview::query(...)`, and `ResourceStreamer::material_management_query(...)` apply status/issue-kind/text filters, sort the filtered rows, compute summary/status buckets and issue-row totals over the full filtered set, and then page the compact overview records. Issue-kind filtering is performed before sorting and paging, so validation, fallback, and diagnostic issue buckets can be queried directly while summary, status indexes, issue indexes, and table pages stay scoped to the same filtered collection. `RenderMaterialManagementRecordSet::query_selection(...)` and `ResourceStreamer::material_management_query_selection(...)` reuse that same query path, then select the full records for the current page ids in page display order. `RenderMaterialManagementQuery::state()` derives the stable control snapshot from the same query object, `has_active_filters()` and `is_paged()` expose the two common table-control predicates, `clear_filters()` drops status/issue/text filters while preserving sort and page, and `first_page_query()` resets only the page offset while preserving the current page size and active filters. `RenderMaterialManagementQuery::active_filters()` and `RenderMaterialManagementQueryState::active_filters()` expose displayable status, issue-kind, and text filter rows. Removing one active filter clears only that filter, preserves the other filters and sort order, preserves the current page size, and resets the offset to zero so a stale offset cannot point outside the newly filtered set. `RenderMaterialManagementQueryResult::state(...)` derives result chrome from the query/result pair, and `RenderMaterialManagementQuerySelection::result_state()` exposes the same classifier for the common page-plus-detail read. `RenderMaterialManagementPageInfo::{next_page_request, previous_page_request}` derive safe adjacent page requests from the returned page metadata, and `RenderMaterialManagementQuery::{next_page_query, previous_page_query}` copies those requests back onto the original filter/sort contract so panels do not have to recompute offsets or risk dropping active filters. `RenderMaterialManagementPageWindow` plus `PageInfo::window()`, `display_start_index()`, `display_end_index()`, `current_page_number()`, and `total_page_count()` provide derived table chrome values for "rows N-M" and "page X of Y" displays without making UI code reinterpret offsets. Unbounded pages and zero-sized pages intentionally produce no navigation request or page-number data, avoiding UI loops around `limit = None` or `limit = 0`; explicit offset-only or zero-limit requests still count as paged query controls in `RenderMaterialManagementQueryState`, while the result state distinguishes an intentional zero-limit empty page from an out-of-range offset or a filter that returned no rows. This keeps filter badges, empty states, page tables, selected-detail records, and pagination controls consistent without requiring editor/runtime panels to re-scan `MaterialRuntime` or reinterpret page-local rows as collection totals.

Query facets use those same filtered indexes. `RenderMaterialManagementQueryResult::facets(...)` and `RenderMaterialManagementQuerySelection::result_facets()` expose status and issue-kind rows with counts from the filtered set before pagination, while each facet's select-query resets the page offset to zero and keeps the current page size. This keeps facet menus, active filter chips, table empty states, and pagination controls aligned with the same query semantics.

Management selection uses the full record set instead of compact overview rows. `management/selection.rs` owns `RenderMaterialManagementSelection`, duplicate-collapsed request counting, missing-id tracking, and selected-record summary/index derivation. `RenderMaterialManagementRecordSet::select(...)` and `ResourceStreamer::material_management_selection(...)` are intended for selection/detail panels that need issue-state and prepared-state rows for one or more ids after a query page has identified them. The selected-record summary includes the same status buckets and issue-row totals scoped only to records that were actually found, while missing ids are explicit so callers can clear stale selections or show that the selected material is no longer prepared.

Mesh pipeline creation consumes `PipelineKey`: `double_sided` disables back-face culling, `alpha_blend` controls transparent blend/depth-write behavior, `alpha_mask` and cutoff bits keep mask variants distinct, and PBR texture slots select authored material variants. Alpha-mask materials are not treated as transparent; only blend mode reports `PipelineKey::is_transparent()`. The mesh and deferred geometry pipeline layouts now include material group 3, and mesh draws bind the prepared material uniform after model and texture bindings. The fallback and deferred geometry shaders declare the neutral group-3 material uniform but do not yet use it for shading until shader reflection/property binding semantics are expanded.

`ResourceStreamer::ensure_scene_resources(...)` counts prepared materials and folds each material readiness summary into renderer-facing counters. `RenderStats` exposes `last_material_count`, `last_material_ready_count`, `last_material_fallback_count`, `last_material_validation_error_count`, and `last_material_diagnostic_count` so submit tests and tools can distinguish ready materials, fallback/degraded materials, blocking validation errors, and non-blocking import/authoring notes without reparsing the full readiness report.

The `.zmaterial` assetization lane now owns material source parsing: built-in material import accepts `.zmaterial`, shader-owned overrides, and texture-slot references. This render-framework module still owns neutral runtime readiness and PBR descriptor projection only; material-editor authoring, automatic shader reflection, and a full physically based shader rewrite remain outside this document's runtime material contract scope.
