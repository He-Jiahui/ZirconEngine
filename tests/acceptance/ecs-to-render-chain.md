# ECS To Render Chain

## Scope
- M0/M1 evidence for the ECS-to-render-chain milestone, typed ECS M2 evidence, transform/active dirty-state M2 evidence, canonical render-extract M3 evidence, the first SystemParam/Commands/change-detection slice, plus the first M4 editor edit-mode projection slice.
- Affected layers: `zircon_runtime::scene::ecs`, World typed component/resource/bundle APIs, SystemParam access state, deferred Commands, component/resource change ticks, `World` dirty derived-state support, `PostUpdate` derived-state propagation, `RenderExtract` preparation, `WorldDriver` schedule execution, scene runtime hooks, scene render extract, the renderer-facing `RenderFrameExtract` baseline, and `zircon_editor::scene::viewport` edit-mode projection DTOs.

## Baseline
- Earlier baseline attempts were blocked by a partial scene ECS cutover: unresolved `scene::ecs`, `SystemStage`, and `Schedule` imports from the deleted `scene::components::schedule` path.
- Earlier continuation attempts also hit transient unrelated compile blockers from other active lanes, including asset facade generic derives, native plugin ABI callback mutability, active asset importer M3 hard-cutover drift, and active UI focus/navigation compile drift.
- Before final M2 validation, the scene runner was corrected to flush successful hook mutations at stage completion, lower ECS component storage was corrected to reject removal type mismatches through the same registered component-type gate used by insertion, stale asset importer outcome consumers were cut to the explicit-locator multi-entry contract, and narrow UI focus compile blockers were repaired so the scene tests could execute.
- Existing known repository baseline: many unrelated UI/editor/plugin/files are dirty in the shared `main` checkout. This acceptance record covers narrowed `zircon_runtime` scene/graphics tests plus an attempted narrowed editor projection validation, not full workspace acceptance.

## Results
- Passed checks: earlier scoped runtime library type check, earlier focused typed ECS API tests, earlier focused M2 dirty-only mutator regression, fresh focused render-extract M2 regression, fresh M3 direct render-frame extract regression, fresh M3 structural hard-cutover guard, fresh M3 scene-produced M5 flagship sideband regression, fresh `graphics::tests`, and the fresh SystemParam/Commands/change-detection checks including `scene::tests` passed.
- Failed or blocked checks: the latest editor projection command has not been rerun in this slice, full workspace build/test plus plugin workspace validation remain unclaimed, and the repo-local default Cargo `target` showed a dep-info write race during validation.
- Fixes made in response: `SceneScheduleRunner` now flushes successful hook-stage mutations after defer mode is disabled, `ComponentStorage::remove<T>` validates the registered component Rust type before removing data, asset importers/tests use explicit root locators for `AssetImportOutcome::new(locator, asset)`, structured asset-label errors now use `source_uri` fields, the active UI focus/input drift was minimally corrected so crate lib-test validation could compile, and the M3 direct render-frame extract test fixture now updates transform before switching a node to static mobility.
- Scene ECS schedule support now has documented resource/event stores, built-in system descriptors, `PostUpdate` hard cutover, `RenderExtract` stage execution, and dirty derived-state flushing behavior.
- Scene ECS M2 typed API now has documented typed `Component`, `Resource`, and `Bundle` contracts, component/resource registries, fixed component setter convergence, dynamic plugin component presence, and runtime-only typed ECS state that stays out of scene project serialization.
- Scene ECS now has documented `SystemState`, `SystemParam`, `Commands`, `apply_deferred`, `ChangeTick`, `ComponentTicks`, `Added<T>`, and `Changed<T>` foundations for the M4/M5 roadmap overlap.
- Scene ECS M2 dirty-state cutover now documents dirty-only mutators, direct current read projection, schedule-maintained node/world-matrix caches, `node_records()` for current direct projection, and owned `find_node()` call sites.
- Scene render extract documentation now records the prepared mutable path, source-preserving clone path, dirty-state flush behavior, M3 direct `RenderFrameExtract` section population, and the rule that `RenderFrameExtract::from_snapshot(...)` is only a preview/roundtrip/legacy adapter, not the scene producer path.
- Editor M4 projection now has a folder-backed `edit_mode_projection` module that derives hierarchy rows, inspector fields, toolbar state, and stats from runtime `World` while keeping selection/tools/overlays editor-owned.

## Acceptance Decision
- Accepted for the narrowed M0/M1 runtime scope, typed ECS M2 slice, transform/active dirty-state M2 slice, canonical render-extract M3 slice, and focused SystemParam/Commands/change-detection slice: the listed `zircon_runtime` checks passed with `--locked` on 2026-05-08.
- Editor M4 projection is implemented and documented, but its focused Cargo validation was not rerun after the runtime M2 pass.
- Blocked for full workspace acceptance: workspace build/test and plugin workspace validation were not run in this slice, and the checkout contains many unrelated dirty files owned by active parallel sessions.
- Remaining risks: M4+ material, postprocess/SRP, renderer submit, VG, and GI depth remain open. Full CI-parity validation remains required before later milestone promotion.
